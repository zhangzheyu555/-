# 功能更新部署流程

**适用环境**：腾讯云生产服务器（OpenCloudOS 9.6，`/root/store-profit/`，域名 `ruguotea.cn`）
**本地环境**：Mac（已装 JDK 21 + Maven + Node + Docker）
**最后更新**：2026-07-18

本文档覆盖三类日常更新：**后端代码更新**、**前端代码更新**、**数据库结构变更**。每次更新按对应章节操作即可。

---

## 通用前提

| 项 | 值 |
| --- | --- |
| 本地项目目录 | `~/Downloads/--codex-deepseek-assistant-latest` |
| 服务器部署目录 | `/root/store-profit/`（用宝塔面板管理文件、宝塔终端执行命令） |
| 服务器登录方式 | 宝塔面板（浏览器）→ 「终端」 |
| 上传文件方式 | 宝塔面板 → 文件 → 上传（单文件 ≤ 500MB） |
| 服务管理 | 在 `/root/store-profit/` 下执行 `docker compose ...` |

**更新前必做**：

1. 确认本地代码改动已完成、本地测试通过。
2. 确认服务器 6 个容器当前健康（`docker compose ps` 全部 `Up (healthy)`），否则先排查现有问题。
3. 关键数据更新前**备份**（至少备份数据库，见文末「备份」一节）。

## 一、后端代码更新

**触发场景**：改了 `backend/` 下的 Java 代码、接口逻辑、配置文件（`application.yml`）等。

### 流程

1. **本地打包 jar**

   ```bash
   cd ~/Downloads/--codex-deepseek-assistant-latest/backend
   mvn -q -DskipTests package
   ```

   产物：`backend/target/store-profit-backend-0.1.0-SNAPSHOT.jar`（约 86MB）。
   打包成功无输出即 OK；若报错，先在本地解决再继续。

2. **打包上传**

   ```bash
   cd ~/Downloads/--codex-deepseek-assistant-latest
   tar -czf /tmp/backend-update.tar.gz backend/target/store-profit-backend-0.1.0-SNAPSHOT.jar
   ```

   用宝塔上传 `/tmp/backend-update.tar.gz` 到服务器 `/root/store-profit/`。

3. **服务器解压 + 重建镜像 + 重启**

   ```bash
   cd /root/store-profit
   tar -xzf backend-update.tar.gz
   docker compose build backend
   docker compose up -d backend
   sleep 30
   docker compose ps backend
   ```

   期望 `Up (healthy)`。若 `Restarting` 或 `Exit`，看日志：`docker compose logs backend --tail=50`。

4. **验证**

   ```bash
   curl -s https://ruguotea.cn/api/health
   ```

   返回 `{"success":true,...,"status":"UP"}` 即正常。然后用浏览器登录 `https://ruguotea.cn/admin/` 验证改动生效。

### 常见坑

- **改了 `application.yml` 但没生效**：配置改动必须重新打包 jar（步骤 1），只重启容器不会读新配置。
- **改了 CORS 白名单**：不用重新打包，直接改服务器 `.env` 的 `APP_CORS_ALLOWED_ORIGINS` 再 `docker compose up -d backend` 即可。
- **Flyway 启动失败**：见「三、数据库结构变更」。

## 二、前端代码更新

**触发场景**：改了 `frontend-vue/` 下的 Vue 代码、页面、样式、路由等。

### 流程

1. **本地构建 dist**

   ```bash
   cd ~/Downloads/--codex-deepseek-assistant-latest/frontend-vue
   npx vite build --base=/admin/
   ```

   ⚠️ **必须带 `--base=/admin/`**（匹配 gateway 路由），否则上线后资源 404、页面白屏。
   产物在 `frontend-vue/dist/`（约 2MB）。构建成功输出 `✓ built in xxxs`。

2. **打包上传**

   ```bash
   cd ~/Downloads/--codex-deepseek-assistant-latest
   tar -czf /tmp/frontend-update.tar.gz -C frontend-vue dist
   ```

   用宝塔上传 `/tmp/frontend-update.tar.gz` 到服务器 `/root/store-profit/`（约 1MB，秒传）。

3. **服务器替换 dist + 重建镜像 + 重启**

   ```bash
   cd /root/store-profit
   # 备份旧 dist（便于回退）
   rm -rf frontend-vue/dist.bak && mv frontend-vue/dist frontend-vue/dist.bak
   # 解压新 dist
   tar -xzf frontend-update.tar.gz -C frontend-vue/
   # 重建镜像 + 重启
   docker compose build frontend
   docker compose up -d frontend
   sleep 10
   docker compose ps frontend
   ```

4. **验证**

   浏览器打开 `https://ruguotea.cn/admin/`，**按 `Ctrl + Shift + R` 硬刷新**（清旧缓存），确认改动生效。

### 常见坑

- **页面白屏**：99% 是忘了带 `--base=/admin/`，或路由 base 配置被改回 `createWebHistory()`。检查 `src/router/index.ts` 应为 `createWebHistory(import.meta.env.BASE_URL)`。
- **改了没生效**：浏览器缓存，必须硬刷新（`Ctrl+Shift+R`）或清缓存。
- **资源 404**：确认新 dist 里的 `index.html` 引用的 JS 路径都以 `/admin/assets/` 开头。

## 三、数据库结构变更（新增 Flyway 迁移）

**触发场景**：新增了 `backend/src/main/resources/db/migration/VXX__xxx.sql` 迁移脚本。

### ⚠️ 重要前提：当前版本差异

| 项 | 版本 |
| --- | --- |
| 本地代码的 Flyway 迁移 | 到 **V62** |
| 服务器数据库已执行的迁移 | 到 **V68**（首次部署时导入了演示数据包） |

**当前数据库比代码新 6 个版本**。这意味着：
- 新增迁移脚本时，**版本号必须 > V68**（如 `V69__...`），不能用 V63~V68，否则 Flyway 会报版本冲突。
- 建议尽快把服务器上的 V63~V68 迁移脚本补回本地代码库（从演示数据包提取或重新编写），消除版本差异。

### 流程

1. **新增迁移脚本**

   在 `backend/src/main/resources/db/migration/` 下新增文件，命名规则：
   ```
   V<版本号>__<简短描述>.sql
   ```
   版本号连续递增、**大于当前服务器数据库最大版本（V68）**。例如 `V69__add_xxx_column.sql`。

   ⚠️ 迁移脚本**一经执行不可修改**（Flyway 校验 checksum），只能新增版本修正。详见 `AGENTS.md`。

2. **按「后端代码更新」流程操作**

   数据库迁移由 backend 启动时自动执行（`application.yml` 的 `spring.flyway.enabled=true`）。所以流程是：
   - 把新迁移脚本随 jar 一起打包（步骤同后端更新）
   - 上传、重建镜像、重启 backend
   - backend 启动时 Flyway 自动检测并执行新迁移

3. **验证迁移成功**

   ```bash
   cd /root/store-profit
   # 看 backend 启动日志里的 Flyway 输出
   docker compose logs backend --tail=80 | grep -i flyway
   ```

   期望看到 `Successfully applied V69__xxx` 或 `Current version of schema ... : 69`。
   若看到 `Validate failed` 或 `Migration ... not resolved`，说明版本号冲突，见下方排错。

4. **验证数据**

   ```bash
   docker exec codex-deepseek-assistant-mysql \
     sh -c 'exec mysql -uroot -p"$MYSQL_ROOT_PASSWORD" -e "use store_profit_qa; <你的验证SQL>"' 2>/dev/null
   ```

### Flyway 常见错误

- **`Validate failed: Migration checksum mismatch`**：某个已执行的迁移脚本被修改了。解决：恢复脚本原内容，或写新版本修正，**不要改老脚本**。
- **`Migration ... was applied but not resolved`**：数据库里有某个版本的记录，但代码里找不到对应脚本（当前服务器 V63~V68 就是这种情况，本地代码缺这些脚本）。目前 backend `enabled=true` 但能正常启动，说明 Flyway 暂时容忍了；新增迁移时只要版本号 > 68 就不会受影响。
- **`Detected resolved migration not applied to database`**：代码里有新迁移但数据库还没执行。正常情况，backend 启动会自动执行。
- **启动卡死或失败**：临时设置 `spring.flyway.validate-on-migrate=false` 跳过校验（在 backend 环境变量加 `SPRING_FLYWAY_VALIDATE_ON_MIGRATE: "false"`），但这是应急手段，根因还是要修正版本。

## 四、多模块同时更新

如果一次改动涉及**后端 + 前端 + 数据库**（常见于功能迭代），按顺序操作：

1. 数据库迁移脚本先行（写好 V69__xxx.sql）
2. 后端打包（包含新迁移脚本）→ 上传 → 重启 backend → 确认 Flyway 执行成功
3. 前端打包 → 上传 → 重启 frontend → 硬刷新验证

**顺序很重要**：先迁移数据库再重启后端，避免后端代码读到不存在的列/表。前端最后更新，避免前端调用了后端还没提供的新接口。

## 五、备份与回退

### 备份数据库（更新前必做）

```bash
cd /root/store-profit
# 导出全库到 backup 目录
mkdir -p backup
docker exec codex-deepseek-assistant-mysql \
  sh -c 'exec mysqldump -uroot -p"$MYSQL_ROOT_PASSWORD" --single-transaction --default-character-set=utf8mb4 store_profit_qa' \
  > backup/store_profit_qa_$(date +%Y%m%d_%H%M).sql
ls -lh backup/
```

### 回退后端

```bash
cd /root/store-profit
# 如果之前打包时备份了旧 jar, 替换回去重建
cp backup/store-profit-backend-旧版本.jar backend/target/store-profit-backend-0.1.0-SNAPSHOT.jar
docker compose build backend
docker compose up -d backend
```

### 回退前端

```bash
cd /root/store-profit
# 恢复之前备份的 dist
rm -rf frontend-vue/dist && mv frontend-vue/dist.bak frontend-vue/dist
docker compose build frontend
docker compose up -d frontend
```

### 回退数据库

⚠️ 数据库回退风险高，只用于严重事故。用更新前的备份恢复：

```bash
cd /root/store-profit
docker exec -i codex-deepseek-assistant-mysql \
  sh -c 'exec mysql -uroot -p"$MYSQL_ROOT_PASSWORD" --default-character-set=utf8mb4' \
  < backup/store_profit_qa_<时间戳>.sql
```

⚠️ 注意：`mysqldump` 备份恢复会覆盖现有数据，恢复前确认备份文件是更新前那一刻的。

## 六、常用运维命令速查

在 `/root/store-profit/` 下执行：

```bash
docker compose ps                      # 看所有服务状态
docker compose logs -f backend         # 实时看后端日志（Ctrl+C 退出）
docker compose logs -f gateway         # 实时看 nginx 日志
docker compose restart backend         # 重启单个服务（不重建镜像）
docker compose up -d backend           # 应用 .env 或镜像变更（重建容器）
docker compose down                    # 停止全部
docker compose up -d                   # 启动全部
docker compose exec mysql mysql -uroot -p   # 进入 MySQL 命令行
```

**修改 `.env` 后**（如改 CORS、改密码）：

```bash
docker compose up -d backend           # 让 backend 重新读取环境变量
```

**修改 nginx 配置后**（`deploy/nginx-unified.conf`）：

```bash
docker compose restart gateway
```

每次更新上线前，对照检查：

- [ ] 本地代码改动已测试通过

- [ ] 新增 Flyway 迁移版本号 > V68
- [ ] 前端构建带了 `--base=/admin/`

- [ ] 小程序如有改动，服务器域名配置已更新并重新发版
