# 内部试用逐关测试总计划

状态：待执行  
适用对象：准备交付给客户内部试用人员的候选版本  
放行结论：只有全部关卡均为 **PASS**，才可标记为 `INTERNAL-PILOT GO`。

## 1. 测试目标与硬性规则

本计划覆盖当前桌面 Web 正式入口 `frontend-vue`、`backend` 中登记的 **45 项正式业务能力**，并覆盖所有正式角色、权限边界、跨角色闭环、附件/导出、外部依赖和桌面浏览器。

本批次**不包含移动 H5、微信、Android App 或 iOS App**；它们不作为内部试用放行条件，也不得被写入本批次的通过结论。

正式角色仅为：`BOSS`、`FINANCE`、`WAREHOUSE`、`STORE_MANAGER`、`SUPERVISOR`、`EMPLOYEE`。督导 `SUPERVISOR` 全面接管原 `OPERATIONS` 的平台配置、培训考试、盘存和运营工作台功能，并保留巡检、整改和日报损复核职责。`ADMIN`、`OWNER`、`OPS`、`OPERATIONS` 均为历史兼容输入，不创建独立业务测试角色。

### 逐关放行规则

1. 测试严格按 G0 → G7 顺序执行；**同一时刻只执行当前关卡**。
2. 当前关卡任一用例为 `FAIL`、`BLOCKED` 或证据缺失，即停止后续关卡；登记缺陷，修复后从该关卡首例重新执行。
3. 禁止把 Mock、单元测试、页面可打开、健康检查或口头确认当作真实 QA 通过证据。
4. 所有写入只能发生在可重置的 QA MySQL 数据库；数据库名必须含 `qa` 或 `test`，且不得连接生产、客户正式库或本机用途未知的数据库。
5. 所有写入数据统一命名为 `E2E-<批次>-<角色>-<模块>-<序号>`；测试结束后清理，或在报告中列出未清理主键、状态和负责人。
6. 外部平台、AI、YOLO、对象存储、视频和 PDF 均须使用测试配置。没有受控测试配置时，状态只能是 `BLOCKED`，不得写成通过。
7. P0/P1 缺陷为零；P2 缺陷必须有产品负责人确认的后续版本和不影响试用核心闭环的说明。

## 4. G1：构建与基础安全

按以下顺序执行，任一失败停止：

1. 后端全量测试：`mvn -q test`。
2. 在隔离工作副本或候选目录完成：`mvn -q -DskipTests package`。
3. 前端执行类型检查和生产构建：`npm run type-check`（如存在）及 `npm run build`。
4. 空 QA MySQL 执行 Flyway 至候选最新版本；再对脱敏升级库执行升级并比对核心表计数、权限、账户密码哈希与附件元数据。
5. 启动候选服务，验证 health/readiness、数据库版本、前端静态资源和 API 基址一致。
6. 未登录访问每个一级业务域至少一个 API，全部返回 401；DENY-A 访问每个一级业务域写接口，全部返回 403 或明确业务拒绝。
7. 扫描候选包、构建日志和前端产物，不得包含密码、Token、API Key、客户数据或测试账号明文。

## 5. G2：六角色公共与权限测试

对每一个正式角色重复以下步骤；一个角色未通过，不进入下一个角色。

1. 登录 → 验证默认工作台、姓名、角色、可访问门店/仓库范围。
2. 逐个检查侧边菜单：允许菜单可打开；不允许菜单不可见。
3. 复制一个不允许页面 URL 直接访问；前端必须阻止或跳转，后端接口必须返回 403。
4. 修改门店 ID、仓库 ID、员工 ID、附件 ID、导出筛选参数；不得读写其他范围数据。
5. 测试退出、刷新、会话过期、权限变更后重新登录；不得残留旧权限或业务数据。
6. 检查支持的桌面分辨率 1366×768、1440×900、1920×1080：无关键内容遮挡或整页横向滚动，按钮可点，错误为中文业务提示。
7. 对本角色一个读取、一个写入/审批/导出（无写权限时为拒绝）执行操作日志验证。

### STORE-01 专项：月度 Excel 导出

1. 店长 A 导出门店 A 的指定月份；文件为 `.xlsx`，仅含业务数据。
2. 用 Apache POI 打开文件，验证“每日汇总”“报损明细”两个 Sheet、日期/金额/数量格式、筛选、冻结表头、文件名及数据范围。
3. 验证未报日期在每日汇总中体现；报损明细包含品类、数量、单位、单价、金额、原因、状态、提交和复核信息。
4. 验证文件没有图片、附件二进制、照片 URL、ZIP MIME 或旧“导出照片包”入口。
5. 店长 A 尝试导出门店 B，SUPERVISOR 尝试导出未授权门店，均须 403；FINANCE/BOSS 的全范围导出仅包含本租户数据。
6. 验证 `daily_loss_export` 操作日志包含导出人、月份、门店范围和行数，但不记录报损明细或照片内容。

## 7. G4：跨角色闭环测试

下列闭环必须按角色顺序串行操作；每一步立即查询业务表、库存/金额/状态与 `operation_log`，并对关键提交重复一次验证幂等。

| 编号 | 角色顺序 | 测试步骤与核心断言 |
| --- | --- | --- |
| FLOW-01 | SM-A → WH-A → SM-A | 叫货 → 审核/发货 → 收货；状态、两端库存、流水、收货仓快照正确；重复收货不重复入库。 |
| FLOW-02 | WH-A → BOSS-A/FIN-A | 创建采购 → 审批/入库 → 查看库存/预警/单据；金额、库存、PDF 与日志一致。 |
| FLOW-03 | 区域仓 WH-A → 总仓 WH-A → 区域仓 WH-A | 补货/调拨申请 → 审批 → 发货 → 收货 → 退货；路线、直属关系、在途和两端守恒正确。 |
| FLOW-04 | FIN-A → BOSS-A → SM-A | 录入或导入月度经营数据 → 老板全局查看 → 店长只读本店；BigDecimal 计算和导出范围一致。 |
| FLOW-05 | SM-A → SUP-A → SM-A → SUP-A/BOSS-A | 日报损提交 → 复核 → 月度 Excel 导出；或巡检问题 → 整改 → 复核；附件、状态、库存影响和待办回写正确。 |
| FLOW-06 | SUP-A → EMP-A → BOSS-A/SM-A | 发布培训/考试 → 学习、答题、交卷 → 查看成绩与进度；重复交卷不重复记录。 |
| FLOW-07 | FIN-A/SM-A/EMP-A/SUP-A | 分别验证经营助手、员工助手和 AI/YOLO：正常、未配置、超时、拒绝敏感信息、人工接管五种状态，不得伪造成功或泄露上下文。 |

## 8. G5：外部能力、附件与故障测试

1. 企迈 QMAI-01 至 QMAI-04 必须在 Mock 服务或官方沙箱中完整执行查询、筛选、导出、失败重试和品牌范围拒绝；没有测试凭据即阻断，不可跳过。
2. 附件对报销、日报损、巡检、退货分别执行上传、预览、下载、错误格式/大小、跨店/跨员工 ID 访问、文件缺失和重复上传测试。
3. PDF 对采购入库、出库、退货单执行下载、A4 黑白可读、数据一致、无权限拒绝和日志测试。
4. AI/YOLO 分别注入未配置、超时、5xx、隐私/敏感信息输入；必须给出中文可恢复提示，不能自动扣分、写入伪结果或泄露密钥。
5. 对象存储/文件服务短暂不可用时，验证业务事务不会出现“记录成功但附件假成功”的不一致状态。

## 9. G6：安全、并发、恢复与性能

1. 对登录、附件、导出、资源 ID、CORS、XSS、伪装文件、Token 失效、权限修改后的旧会话执行安全测试。
2. 对发货、收货、采购入库、调拨、退货、日报损复核、报销审批、工资状态、考试交卷分别并发提交至少两次；断言只有一次状态迁移和一次库存/金额/成绩写入。
3. 执行前端刷新、后端重启、数据库短暂不可用、对象存储故障、AI/YOLO 故障和网络切换；不允许部分提交、重复扣库存或丢失已选择文件。
4. 记录候选环境下工作台、列表、筛选、提交、导出、PDF、上传、视频首帧、AI 调用耗时；超过约定 SLA 的项目不能标 PASS。
5. 验证应用回滚和数据库恢复演练；必须明确应用回滚不等于数据库回滚。

### 放行判定

可签署 `INTERNAL-PILOT GO` 的条件：

- G0 至 G7 全部 PASS；45 项功能覆盖率为 45/45；6 个正式角色覆盖率为 6/6；历史 OPERATIONS 账号迁移/停用验证通过。
- QMAI、AI/YOLO、附件、导出、PDF、视频等外部或文件能力均有真实 QA 证据，不能以“未配置”作为通过。
- 无 P0/P1 缺陷；P2 均已记录且不阻断指定内部试用流程。
- 每条跨角色闭环的业务数据、状态、金额/库存/成绩、附件和审计日志均一致。
- 客户内部试用仅使用新建、最小权限账号；先在一个受控门店/仓库范围小规模启用，并保留问题反馈渠道与回退方案。

任一条件不满足，结论只能为 `NO-GO` 或 `GO WITH BLOCKERS`，不得交付客户内部人员试用。
