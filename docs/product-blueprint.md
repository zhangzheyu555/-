# AI Profit OS 产品蓝图

## 目标

将系统收敛为可持续开发和上线的多门店经营管理系统。核心闭环是：门店数据录入 -> 后端利润计算 -> 异常识别 -> 待办分派 -> 业务中心处理 -> 复核 -> 操作日志。

## 正式技术边界

| 层级 | 正式实现 | 说明 |
| --- | --- | --- |
| 前端 | `frontend-vue` | Vue 3、TypeScript、Pinia、Vue Router |
| 后端 | `backend` | Spring Boot、JdbcTemplate、统一 `ApiResponse` |
| 数据 | MySQL + Flyway | 所有真实业务数据必须持久化 |
| 历史入口 | `/legacy` | 旧 HTML 只读回看与视觉参考 |

`web`、`runtime-static`、顶层 `index.html` 和 `database.js` 不再承载新功能。

## 信息架构

| 一级中心 | 页面与能力 | 主要角色 |
| --- | --- | --- |
| 工作台 | 今日待办、风险提醒、待复核事项 | 全角色，内容按权限过滤 |
| 经营中心 | 利润概览、利润表、门店详情、数据录入 | 老板、财务、店长 |
| 财务中心 | 报销、工资、数据核对、数据导出 | 老板、财务；店长仅自店报销 |
| 执行中心 | 督导巡店、仓库、培训、考试、盘存 | 督导、仓库、运营、店长 |
| 管理中心 | 门店、员工、账号权限、平台配置、操作日志 | 系统管理员、老板，部分运营 |

## 核心业务闭环

1. 财务或授权店长录入门店月度经营数据，或者通过 Excel/CSV 导入并确认。
2. 后端以 `BigDecimal` 计算收入、成本、毛利、费用、净利润、净利率和成本占比。
3. 后端根据缺数、亏损、成本超阈值、营收下降、待审报销和待审工资生成待办。
4. 待办仅承担提醒、状态、责任人与跳转；报销、巡检、叫货等复杂操作在对应业务中心完成。
5. 状态变化、审批、附件、导出、权限拒绝均写入操作日志。
6. 已完成事项可以查看，不允许绕过复核任意改写。

## 状态约束

待办统一使用后端枚举：`PENDING`、`IN_PROGRESS`、`PENDING_REVIEW`、`COMPLETED`、`REJECTED`。前端只负责显示中文标签，不拼接新的状态字符串。

## 非目标

- 不再把旧版 `database.js` 当作运行时数据源。
- 不使用 `localStorage` 保存经营、报销、巡检、库存、工资或待办数据。
- 不在平台接口失败时伪造营业额、订单或成功状态。
- 不在前端暴露数据库密码、平台密钥或 AI 密钥。
