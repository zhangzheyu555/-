# AI Profit OS 全角色上线验收测试设计

状态：预发布验收设计，尚未得出“全部功能可上线”的结论  
适用版本：桌面 Vue3、Spring Boot 后端、UniApp H5/微信小程序/Android/iOS 候选端  
最后更新：2026-07-15

## 1. 验收目标与结论标准

本设计验证的不是“页面能打开”，而是以下四件事同时成立：

1. 每个正式角色进入正确的工作台，只看到并能操作本角色被授权的功能。
2. 一个业务动作从发起、处理、交接、确认到日志追溯形成闭环，金额、库存、考试成绩和巡检结论不重复、不丢失、不越权。
3. 未登录、错误角色、跨门店/跨仓、失效附件、重复提交、弱网和上游 AI 服务异常都被安全处理，并以中文业务提示呈现。
4. 同一业务规则在桌面端和已规划的移动端保持一致；移动端只承担一期允许的能力，复杂操作必须明确引导回桌面端。

只有所有 P0 门禁通过、P1 无未豁免缺陷、写入数据已清理或可追溯、证据已归档，才能签署“候选版可上线”。构建成功、健康检查为 UP、或某个角色可登录，都不能单独代表验收通过。

## 2. 真实角色与工作台范围

系统当前正式角色只有六类。`SUPERVISOR` 与 `OPS` 是兼容别名，均应按 `OPERATIONS` 进行验收；不能虚构第七套督导登录账号。

| 正式角色 | 桌面默认工作台 | 移动一期定位 | 必须验证的结果 |
| --- | --- | --- | --- |
| `BOSS` | `/boss` | 经营与考试只读摘要 | 全局风险可见、数据范围正确、复杂业务跳转到业务中心、全局权限与日志可追溯 |
| `FINANCE` | `/finance` | 经营只读摘要 | 利润、月度录入/导入、报销、工资、导出按权限完成，计算和导出范围正确 |
| `WAREHOUSE` | `/warehouse` | 无移动一期菜单 | 物料、采购入库、叫货、发货、调拨、退货、预警、单据和库存账实闭环 |
| `STORE_MANAGER` | `/store` | 库存、叫货、收货 | 仅本店可见；可叫货和确认收货；不得进入经营数据录入、总仓配置或其他门店数据 |
| `OPERATIONS` | `/operations` | 巡检、拍照、AI 建议、整改；培训考试管理 | 巡检和培训考试完整闭环，附件与门店范围正确，模型建议必须人工确认 |
| `EMPLOYEE` | `/learn/exams` | 视频学习、考试、员工服务助手 | 只看本人学习、考试和对话；视频、进度、交卷、成绩和助手安全降级有效 |

## 3. 验收环境与测试数据

### 3.1 环境隔离

- 仅使用独立预发布 MySQL 8 数据库或可回滚的脱敏副本；禁止用生产数据库做写入验收。
- 后端、桌面前端、H5 与候选移动 API 必须指向同一候选版本；记录 Git 提交、JAR SHA-256、Flyway 版本、前端构建哈希和 API 基址。
- AI、YOLO、对象存储、PDF、视频和平台回调使用候选凭据；密钥只由部署环境注入，不出现在用例、截图、日志或仓库中。
- 先完成数据库备份/恢复演练和部署回退预案，才允许执行写入验收。

### 3.2 受控账号与数据范围

需提供脱敏、可重置的账号：老板、财务、仓库管理员、运营、员工 A/B、店长 A/B，以及一个无仓库权限/无门店绑定的拒绝账号。仓库管理员至少覆盖荆州总仓和山东分仓；店长 A/B 分属不同门店；员工 A/B 用于“仅本人可见”的反向校验。

所有写入数据统一使用：

```text
E2E-<发布批次>-<角色>-<业务>-<序号>
```

验收前建立活跃/停用物料、可售/不可售库存、两个门店、总仓与分仓、待处理叫货单、可用巡检标准、培训视频、试卷和附件样本。验收后按批次清理，无法清理时在报告中列出主键、状态和负责人。

## 5. 全角色公共验收域

以下项目对六个角色均为必测范围：

- 登录、退出、刷新、会话过期、权限版本变化和重新登录；Token 不得泄露到业务日志。
- 默认工作台、菜单可见性、直达 URL 路由守卫、浏览器返回/刷新和无权限页。
- 无登录返回 401；无角色权限、跨门店、跨仓、跨员工、越权附件均返回 403；前端隐藏菜单不能替代后端判定。
- 当前用户、门店/仓库/品牌筛选范围与接口返回范围一致；禁止通过修改 URL、请求参数或附件 ID 读取他人数据。
- 业务页面无英文技术状态码、无无意义空白、按钮加载/禁用/失败状态明确；桌面、390px、412px、768px 不发生整页横向溢出。
- 关键读取、修改、审批、导入导出、上传下载、打印、拒绝访问均可在操作日志定位到操作者、对象、时间、结果和请求标识。
- 业务数据只来自 API/MySQL；浏览器或 UniApp 端只允许持久化登录会话，不允许持久化库存、利润、巡检、考试、附件或聊天业务记录。

## 6. 角色工作台验收域

### 6.1 老板 `BOSS`

- 经营、利润、风险、待办、门店、仓库、考试汇总、数据健康与操作日志可在全范围内正确加载。
- 今日待办只能提醒、筛选、跳转和关闭，不承载库存、财务或巡检的复杂编辑操作。
- 从风险/待办跳转到利润、仓库、巡检、报销等业务中心时，筛选条件和数据范围正确传递。
- 对账号权限、门店管理、平台配置、全局导出和审计日志进行权限与审计验证。

### 6.2 财务 `FINANCE`

- 利润概览、利润表、单店单月经营录入、月度汇总导入、报销、工资、导出和经营助手按授权范围正确运行。
- 金额、成本、净利润、净利率和导入覆盖确认由后端 `BigDecimal` 计算；重复导入、冲突覆盖、错误文件、空数据和跨店数据应得到明确结果。
- 数据导出包含正确筛选范围和文件内容；导出行为写入操作日志。
- 财务不得取得仓库管理、账号权限、督导巡检管理等未授权入口或 API 数据。

### 6.3 仓库管理员 `WAREHOUSE`

- 总仓/分仓库存、物料档案、物料启停、采购单、采购入库、库存预警、出入库记录、店长叫货、发货、调拨、退货和 PDF 单据形成完整库存链路。
- 停用物料不能新叫货或入库，但历史、库存流水和打印单仍可追溯。
- 叫货/发货/收货/入库/退货/调拨全部测试重复提交、并发库存不足、禁用路线、非直属仓、部分数量、取消/驳回和错误状态转移。
- 入库单、出库单、配送退货单必须由后端 MySQL 数据生成 A4 黑白可打印 PDF，下载后写操作日志。

### 6.4 店长 `STORE_MANAGER`

- 仅展示所属门店经营结果、工资核对、门店详情、巡检记录、培训进度、本店库存、叫货单和收货入口。
- 从本店库存填写多物料数量、提交叫货、查看状态、仓库发货后确认收货；重复确认不得重复增加库存。
- 尝试直达 `/data-entry`、总仓库存、物料档案、采购入库、其他门店详情及跨门店附件时必须被前端拦截并由 API 返回 403。
- 未绑定门店的店长必须进入清晰的无权限/待绑定提示，不得回退到任意门店。

### 6.5 运营 `OPERATIONS`

- 巡检标准、发起巡检、照片/相册选择、附件上传、YOLO/模型建议、人工确认、扣分、整改、复核、历史记录和导出可形成闭环。
- 模型建议只能辅助，最终扣分、条款关联和整改状态必须由业务规则与人工确认决定；模型不可用时可继续人工处理且不伪造结果。
- 历史巡检证据补传必须仅关联历史条款快照，不得重算历史得分或把照片自动绑定到当前标准。
- 培训视频、题库、试卷、发布、阅卷、成绩报表和学习进度统计按运营权限正确运行；不得读取无权限财务数据。

### 6.6 员工 `EMPLOYEE`

- 仅能读取本人的课程、视频播放票据、学习进度、试卷、答题、交卷、成绩、员工服务助手和个人反馈。
- 视频 Range 播放、进度上报、断网重试、重复交卷、成绩回显、票据过期和跨员工访问均需验证。
- 员工服务助手只使用员工助手专用配置；经营/财务问题、附件、订单、地址、明确客户姓名、密钥等内容必须被拒绝或安全引导；上游不可用时禁止发送并提供中文说明。

## 7. 跨角色业务闭环

每条闭环都要在隔离环境用可清理的 `E2E-*` 数据完成，并保留创建、处理、确认、查询、重复提交和操作日志证据。

| 闭环 | 角色顺序 | 必须断言 |
| --- | --- | --- |
| 门店叫货与收货 | 店长 → 仓库 → 店长 | 路线和库存正确；发货后可收货；重复确认不重复入库；跨店/跨仓被拒绝 |
| 采购入库与预警 | 仓库 → 老板/财务只读 | 采购、入库、库存流水、预警、金额和 PDF 可追溯；无权限角色只读或被拒绝 |
| 仓间调拨与退货 | 分仓/总仓仓库管理员 | 路线、直属关系、审批、发货、收货、退货部门与收货仓快照正确；异常状态不可越迁 |
| 月度经营数据 | 财务 → 老板/店长 | 财务录入/导入结果计算一致；老板看全局，店长仅看本店；店长不得录入 |
| 巡检整改 | 运营 → 店长 → 运营 → 老板 | 附件、条款快照、人工确认、整改、复核和风险待办可回溯；历史分数不被补传重算 |
| 培训考试 | 运营 → 员工 → 老板/店长 | 视频、进度、发布、作答、交卷、成绩、报表一致；重复交卷返回原结果 |
| 双助手与 AI 识别 | 员工/财务/店长/运营 | 正常、未配置、授权异常、超时和拒绝敏感内容五类状态可解释；不暴露 URL、Token、模型密钥或业务敏感内容 |

## 8. 三端与设备矩阵

桌面端必须覆盖六个工作台。移动端一期按角色授权覆盖，不把没有移动入口的复杂仓库/财务配置强行塞进手机。

| 容器 | 必测角色与能力 | 重点差异 |
| --- | --- | --- |
| 桌面 Chromium/Edge | 六个角色全部工作台与跨角色闭环 | 大表格、PDF、导入导出、快捷键、菜单与路由守卫 |
| Android Chrome | 店长、运营、员工，财务/老板只读摘要 | 相机/相册降级、弱网、软键盘、触控、H5 视频 Range |
| iPhone Safari | 店长、运营、员工，财务/老板只读摘要 | 安全区、相册权限、Safari 播放与刷新恢复 |
| Android 微信 | 店长、运营、员工 | 合法域名、相机/相册、上传、更新提示、拒绝授权 |
| iPhone 微信 | 店长、运营、员工 | 同上，并验证 iOS 微信上传和视频行为 |
| Android App | 店长、运营、员工 | Camera/VideoPlayer 模块、版本检查、签名候选、网络/权限拒绝 |
| iOS App | 店长、运营、员工 | 相机/相册用途说明、安全区、视频 header、版本检查与证书候选 |

每个容器至少执行：登录、权限菜单、数据范围、弱网、401、403、上传、重复提交、退出登录。真机报告记录设备型号、系统版本、容器版本、网络条件、执行人、时间、截图和结论。

## 9. 非功能、安全与回滚

- 性能：首页、角色工作台、列表筛选、关键提交、文件上传、PDF 下载、视频首帧与 AI 调用分别定义候选环境 SLA；慢请求需记录接口、请求标识、数据规模和链路耗时。
- 安全：测试 XSS、上传格式/尺寸/伪装文件、附件越权、IDOR、暴力登录保护、Token 失效、日志脱敏、CORS、HTTPS、CSP、视频票据和网关查询参数脱敏。
- 可靠性：后端重启、前端刷新、AI/YOLO/对象存储短暂故障、网络切换和重复请求不得造成部分提交、重复扣库存、重复记账或重复交卷。
- 回滚：候选发布前验证数据库备份可恢复、前端可回退、JAR/静态资源哈希可核验；禁止通过修改已执行 Flyway 迁移回滚结构。

## 11. 自动化与人工边界

- Playwright：执行桌面冒烟、菜单、路由、只读页面、响应式、数据范围和显式开启的隔离写入流。
- 后端测试：验证计算、权限、Flyway、异常、幂等、PDF、视频票据和数据契约。
- Kane：在完成登录后生成并保存可执行的浏览器验收场景；当前未登录时不能把手写清单冒充 Kane 产物。
- 人工真机：微信、Android/iOS App、相机/相册/权限、签名、商店审核资料及实际弱网体验。

建议先在当前 Windows 用户完成 Kane OAuth 登录：

```powershell
$env:KANE_CLI_USER_AGENT = 'codex'
kane-cli login --oauth
kane-cli whoami
```

登录成功后，由 Codex 使用 `kane-cli generate --agent` 根据本设计、`AGENTS.md`、移动发布清单和现有 E2E 指南生成用例；再把 Functional 用例保存到隔离测试目录，禁止直接对生产环境执行。

## 13. 上线门禁

### P0：任一项失败即禁止上线

- 候选构建、后端测试、空 MySQL Flyway、健康检查或关键 API 契约失败。
- 任一角色可越权、跨门店/跨仓/跨员工读取或修改数据，或 401/403 不成立。
- 金额、库存、得分、考试成绩、状态机或幂等性错误。
- 关键附件/PDF/视频不可用、下载/上传导致数据丢失、或 AI/YOLO 异常造成自动错误扣分。
- 存在密钥、密码、Token、个人/客户敏感数据泄露，或没有可验证回滚方案。

### P1：必须修复或取得书面豁免

- 单角色工作台无法完成其核心工作流，或中文错误提示不能指导操作。
- 中低风险 UI/响应式缺陷影响完成任务，导出、日志、性能或设备兼容性与约定不符。
- 移动候选未在目标真机验证、Node 20 LTS 未重建、依赖高危漏洞未处置。

### P2：可排入后续迭代

- 不改变业务结果的文案、间距、非关键动效和优化建议，但必须记录产品负责人和计划版本。
