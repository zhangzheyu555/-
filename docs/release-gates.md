# AI Profit OS 上线门禁

## 目的与边界

本门禁用于把一次候选发布固定为可复查的 **QA → RC → 真机验收** 流程。它不授权连接生产、3307、生产账户、签名、商店提交或 QA 整库清理。任何这些操作必须由发布负责人另行明确授权。

候选必须来自已审核、已提交且干净的 Git 提交。禁止从含未提交修改的工作目录构建；禁止 `git reset`、`git clean`，也禁止改写已执行的 Flyway 迁移。

## 不可绕过的 P0 判定

出现下列任一项即为 P0，停止发布：

- 未登录返回非 401，越权或跨门店访问返回非 403；
- 金额、库存、巡检分数、历史快照或标准版本被错误修改或重算；
- 关键附件、PDF、视频 Range、AI/YOLO 正常或降级路径不可用；
- 重复提交、跨角色闭环错误，或操作日志缺失；
- 候选嵌入 `.invalid`、本机或非 HTTPS API 地址、真实密钥、未审核依赖高危项；
- QA/生产数据库范围不明确、无可验证备份/回滚路径；
- 未签名资源被表述为正式 APK/AAB/IPA，或发生任何商店提交。

## 受控流程

1. 发布负责人审核变更后创建提交，记录提交 SHA；从该 SHA 创建独立 worktree，并确认 `git status --porcelain` 为空。
2. 将受控 Node 20 放在 PATH 首位；在该 worktree 中执行 `npm ci`、Vue 类型检查/构建、后端测试/隔离打包、UniApp 四端构建。所有产物写 SHA-256、版本、构建时间和构建环境。
3. 仅在得到书面授权后运行 QA 整库重置脚本。它只接受 loopback、非 3307、`ai_profit_qa_*` 或 `ai_profit_test_*` 以及非 root 用户，先备份并记录 SHA-256。
4. 仅在独立 QA 库上启动候选，先检查 `/api/health`、环境、Flyway、认证和权限；恢复经批准的脱敏基线后，再创建 `E2E_` 门店、夹具和六个角色账号。夹具创建必须有操作日志收据。
5. 使用 Kane 生成并运行桌面角色与跨角色闭环；真实设备按 `docs/device-test-matrix.md` 执行。失败项记录请求状态、请求 ID、日志位置和复现步骤。
6. 所有 P0 为通过、无未处理 P1、存在回滚证据后，才可由发布负责人标记 RC。签名、商店提交和生产切换仍需单独授权。

## 证据目录约定

```text
output/release-evidence/<batch>/
  preflight/        # 提交、锁哈希、环境和阻断项
  qa/               # reset、候选健康、夹具和操作日志收据
  build/            # 每个候选产物和 SHA-256
  inspection/       # 整改/复核接口、权限和历史保护
  mobile/           # H5/微信/Android/iOS 构建清单
  kane/             # 生成的测试、运行 NDJSON、截图和请求标识
  devices/          # 真机矩阵、截图、网络与权限复现
```

证据不得包含令牌、密码、模型密钥、原始敏感聊天内容或生产数据。
