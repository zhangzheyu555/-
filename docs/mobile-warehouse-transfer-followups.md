# 小程序仓间调拨：后端与权限跟进项

## 现象

仓库作业中心的“仓间调拨”页只展示已驳回单据，没有“新建调拨单”表单；已驳回单据也无法再次提交。

## 已核对结论

1. 小程序以接口 `/api/warehouse/transfers/context` 返回的 `route.actions.canCreate` 作为新建调拨路线的唯一依据。若为 `false`，小程序不会把该路线当作可创建路线。
2. 总仓向分仓主动配货时，后端要求当前账号同时具备总仓和目标分仓的数据范围；仅能访问总仓时，路线仍可能返回审批/发货能力，但 `canCreate=false`。
3. 调拨单被驳回后状态为 `REJECTED`。当前后端只允许提交 `DRAFT` 单据，未提供把 `REJECTED` 退回草稿或直接重新提交的接口，因此前端不能伪造“重新提交”按钮。

## 处理建议

### 先检查账号数据范围（无需代码变更）

为需要从总仓向分仓创建调拨单的仓库管理员，配置总仓及目标分仓（例如“山东分仓”）的数据范围，并确认没有对 `warehouse.transfer.request` 设置 DENY 覆盖。重新登录后，接口应返回该路线 `canCreate=true`，小程序才会展示新建调拨表单。

### 如业务要求驳回后可修改重提（需后端改造）

新增受控的“退回草稿/重新提交”流程：

- 仅申请人、且仅 `REJECTED` 状态可发起；必须保留驳回原因和原审批日志。
- 退回草稿后允许修改数量、物料或备注；再次提交时重新执行库存、路线、权限、幂等和版本校验。
- 写入操作日志；不得修改历史审批记录，也不得对未发货调拨产生库存副作用。
- 推荐新增 `POST /api/warehouse/transfers/{id}/reopen`，而不是让现有 `/submit` 接受所有已驳回单据。
