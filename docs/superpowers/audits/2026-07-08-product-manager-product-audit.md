# AI Profit OS 产品审计与优化方案

日期：2026-07-08  
角色视角：产品经理 / 经营系统负责人  
审计对象：门店利润系统 / AI Profit OS / AI 企业经营分析系统  
审计范围：产品定位、角色工作流、前端体验、数据可信度、AI 价值、上线风险、后续路线  
证据来源：项目 README、CURRENT_VERSION、后端接口结构、前端静态页面、已有设计文档、当前本地截图  

## 1. 最终判断

这个项目有真实价值，方向也对：多门店经营不是缺一张报表，而是缺一个能每天提醒人、分派事、追踪处理结果的经营操作系统。

但当前最大问题是：系统仍然像“功能集合”，还没有完全成为“经营闭环”。功能很多，模块也多，但用户每天进来后仍然需要自己理解菜单、判断优先级、确认数据是否可信、再决定该找谁处理。对老板来说，这会降低信任；对员工来说，这会增加使用负担。

最高优先级不是继续加功能，而是先把三个东西做实：

1. 数据可信：真实业务数据只来自 MySQL 结构化表，前端和兼容 KV 不再成为长期数据来源。
2. 工作闭环：每条异常都有负责人、截止时间、处理状态、来源证据和处理结果。
3. 角色首页：每个角色进入系统后先看到“我今天要处理什么”，而不是先看到一堆模块。

一句话：这个产品应该从“门店利润系统”升级为“多门店经营异常处理系统”。利润只是入口，闭环才是价值。

## 3. 产品定位问题

### 3.1 当前定位过宽

现在项目同时在做：

- 利润看板
- 利润表
- 门店详情
- 督导巡店
- AI 助手
- 数据录入
- 报销
- 导出
- 门店管理
- 平台登录
- 仓库
- 运营
- 用户权限
- 员工工资
- 迁移工具
- AI / DeepSeek
- YOLO 巡店识别

这些能力都有价值，但一起暴露时，产品心智会变散。老板很难一句话理解“我为什么每天要打开它”。

### 3.2 建议定位

建议对外和对内统一定位为：

> AI Profit OS 是多门店经营异常处理系统。它每天把利润、费用、巡店、仓库、工资里的异常自动分派给对应角色，并把处理不了的问题上报给老板。

这个定位比“利润系统”更强，因为它不是只展示结果，而是驱动处理。

### 3.3 产品北极星指标

建议不要把“录入了多少数据”当核心指标，而要看：

- 老板进入系统后 30 秒内是否知道今天最重要的 3 件事。
- 每条红色风险是否 100% 有负责人和处理状态。
- 门店、财务、督导、仓库的待办是否能在系统内闭环。
- 数据刷新后，各角色看到的结果是否一致。
- AI 总结是否每句话都能追溯到真实数据来源。

## 4. 主要问题清单

### P0-1：产品叙事不一致，影响信任

问题：

- README 写“门店利润系统”，新文档写“AI Profit OS / AI 企业经营分析系统”。
- README 和 CURRENT_VERSION 里仍出现旧默认密码和本地存储回退描述。
- 当前前端登录页仍叫“门店利润系统”，但系统实际已经包含仓库、巡店、薪资、AI。

影响：

- 用户不知道这是报表工具、管理后台，还是经营操作系统。
- 部署和交付时容易出现“哪个版本才是正式版”的混乱。
- 对老板来说，产品越不自洽，越不敢把真实经营数据交给它。

优化方案：

- 统一产品名：建议使用“AI Profit OS”作为系统名，“门店利润系统”作为旧名或子模块名。
- 更新登录页、副标题、README、CURRENT_VERSION、部署文档。
- 删除或明确标注旧本地存储回退说明。
- 建立“当前正式版本”一页：入口、数据源、默认账号策略、已上线模块、未上线模块。

验收标准：

- 所有文档只保留一个正式主线。
- 登录页、README、设计文档、部署文档对产品名和数据源描述一致。

### P0-2：数据可信度还没有成为产品的一等公民

问题：

- 系统正在从浏览器存储、兼容 KV、结构化 MySQL 之间迁移。
- 后端已经有结构化 API，但前端仍有不少模块围绕旧数据对象运行。
- 用户界面里没有清楚告诉用户：数据来自哪里、是否已迁移、最后更新时间是什么、是否存在未迁移数据。

影响：

- 财务和老板无法判断报表是不是最新。
- 如果数据出错，很难判断是录入错、迁移错、前端缓存错，还是接口口径错。
- AI 总结如果建立在不稳定数据上，会迅速损害信任。

优化方案：

- 新增“数据健康中心”，显示：
  - MySQL 结构化数据状态。
  - 兼容 KV 剩余业务键。
  - 浏览器旧数据是否存在。
  - 最近一次迁移时间和迁移结果。
  - 每个模块的数据来源：结构化 API / 兼容 KV / 未接入。
- 在关键页面展示数据更新时间。
- 所有经营报表、待办、AI 总结必须基于后端结构化服务。

验收标准：

- 老板能看到“当前数据可信状态”。
- 财务能看到哪些门店、月份、模块还未完成迁移。
- 真实业务写入不再进入浏览器本地存储。

### P0-3：角色闭环不完整，待办页还只是入口

问题：

- “今日待办”已经是正确方向，但第一版主要负责展示和跳转。
- 一条待办的生命周期还不完整：谁负责、截止时间、是否已处理、处理证据、处理不了如何上报，这些还没有成为统一模型。
- 财务、督导、仓库上报老板的机制还没有形成统一体验。

影响：

- 用户点进处理页后，仍然可能不知道下一步该做什么。
- 老板看到异常，但不知道这件事现在卡在谁手里。
- 系统无法回答“昨天的 5 个风险处理完了吗”。

优化方案：

- 建立统一待办生命周期：
  - 新建
  - 已分派
  - 处理中
  - 已处理
  - 已驳回
  - 已上报老板
  - 已关闭
- 每条待办必须有：
  - ownerRole
  - ownerUser
  - sourceModule
  - sourceRecordId
  - severity
  - dueAt
  - status
  - actionUrl/actionParams
  - resolutionNote
  - escalationReason
- 今日待办页仍不直接处理复杂业务，但要显示处理状态和负责人。

验收标准：

- 任意一条红色风险都能追溯到来源数据、负责人、处理记录。
- 老板可以筛选“已上报给我但未处理”的事项。
- 财务、督导、仓库上报必须填写原因。

### P0-4：信息架构仍然是模块菜单，不是工作流

问题：

左侧菜单现在有 16 个入口：

- 今日待办
- 利润概览
- 利润表
- 门店详情
- 督导巡店
- 门店助手
- 数据录入
- 报销栏
- 数据导出
- 门店管理
- 操作日志
- 平台登录
- 仓库中心
- 运营中心
- 用户权限
- 员工工资

对老板和一线角色来说，这仍然偏“后台系统”。功能很多，但层级不清楚。

影响：

- 新用户学习成本高。
- 店长、财务、仓库管理员容易进入不相关页面。
- “今日待办”虽然排第一，但后面仍然是大模块堆叠。

优化方案：

按角色分组导航，而不是所有模块平铺：

老板：

- 今日待办
- 经营概览
- 风险追踪
- 门店详情
- AI 经营参谋
- 管理设置

财务：

- 今日待办
- 数据录入
- 报销审核
- 工资核对
- 报表导出

督导：

- 今日待办
- 发起巡店
- 整改跟踪
- 巡店标准

店长：

- 今日待办
- 本店经营
- 巡检整改
- 提交报销
- 门店叫货

仓库：

- 今日待办
- 叫货处理
- 库存预警
- 入库采购
- 物料档案

验收标准：

- 每个角色左侧一级入口不超过 7 个。
- 当前角色看不到明显无关入口。
- 老板可切换完整管理视图，但默认仍是经营工作流。

### P0-5：AI 价值还没有建立“可信解释”

问题：

- 老板首页的 AI 总结方向正确，但现在还容易被用户理解为“AI 写了一段话”。
- AI 总结缺少引用证据、置信边界、可执行建议和处理入口。
- DeepSeek 失败时虽有规则兜底，但用户不一定知道两者区别。

影响：

- 如果 AI 说错一次，老板会怀疑整个系统。
- AI 不能只做文案层，它必须成为“有证据的经营参谋”。

优化方案：

AI 总结改成三段结构：

1. 今天先看什么：最多 3 件事。
2. 为什么：列出数据证据，如门店、月份、净利率、环比变化。
3. 怎么处理：给出下一步入口，跳到利润表、报销、仓库、整改。

每条 AI 建议必须显示：

- 数据来源
- 数据时间
- 使用规则
- 是否为 DeepSeek 生成
- 是否为规则兜底

验收标准：

- AI 总结中的每一句判断都能点开来源数据。
- DeepSeek 不可用时，页面仍能给出结构化规则总结。
- AI 不直接做业务决策，只建议和解释。

### P1-1：老板利润概览还缺少“行动优先级”

问题：

利润概览已经能显示收入、净利、净利率、门店排名和趋势，但它仍然偏报表。老板看到“净利 -14500”后，还需要自己判断：

- 哪个店拖累最多？
- 是收入下滑、成本上升，还是费用异常？
- 该让谁处理？
- 是否已经有人在处理？

优化方案：

在利润概览顶部增加“经营风险解释条”：

- 本月亏损来源 TOP 3。
- 净利率低于目标门店。
- 环比下降最大门店。
- 最可能原因：收入、原材料、人工、租金、维修、推广等。
- 直接生成待办或跳转处理。

验收标准：

- 老板不用看完表格，也能知道本月利润差在哪里。
- 每个风险都能跳转到对应利润表或明细。

### P1-2：报销流程缺少明确状态机

问题：

报销栏目前有待审核、已完成、金额、门店、类别等信息，但流程仍偏简单。

缺失点：

- 付款凭证缩略图不是列表核心信息。
- 驳回原因、补充材料、再次提交不够清晰。
- 财务和老板谁审核，需要规则明确。
- 报销和费用异常没有和利润异常打通。

优化方案：

报销状态建议：

- 草稿
- 已提交
- 财务初审
- 财务驳回
- 老板复核
- 已通过
- 已入账

第一版可以简化为：

- 待审核
- 已驳回
- 已完成

但必须增加：

- 审核人
- 审核时间
- 驳回原因
- 是否已计入利润费用
- 关联月份

验收标准：

- 财务可以筛选“待我处理”。
- 店长能看到为什么被驳回。
- 老板能看到大额或异常报销。

### P1-3：仓库中心承载了太多不同工作

问题：

仓库中心把库存预警、门店叫货、新增物料、采购到货、台账放在一个长页面里。对仓库管理员来说，第一优先级应该是处理叫货单，但截图里“门店叫货表单”和“新增物料表单”抢占了大量首屏注意力。

影响：

- 仓库管理员容易被管理表单干扰。
- 店长进入仓库页时也可能看到过多仓库后台信息。
- 待处理叫货单没有成为页面第一中心。

优化方案：

仓库页按任务重排：

1. 待处理叫货单
2. 缺货 / 临期风险
3. 今日出库记录
4. 采购到货
5. 物料档案

新增物料、采购入库、物料台账放到二级 tab 或折叠区。

验收标准：

- 仓库管理员进入页面后第一眼看到待处理叫货单。
- 店长只看到“提交叫货”和自己的叫货历史。
- 采购/物料维护不会干扰日常叫货处理。

### P1-4：移动端不是现在主线，但店长场景迟早会卡住

问题：

当前明确先保证电脑端，这对老板、财务、仓库可以接受。但店长和督导很可能在手机上使用：

- 店长提交报销。
- 店长看巡检整改。
- 店长提交叫货。
- 督导巡店拍照。

如果移动端长期不补，系统会在一线执行场景断掉。

优化方案：

不要先做全站移动端。先做两个移动优先流程：

1. 店长今日待办
2. 督导巡店拍照 / 整改跟踪

验收标准：

- 店长手机上 3 步内完成报销提交。
- 店长手机上能看到整改要求和截止时间。
- 督导手机上能完成巡店记录和照片上传。

### P1-5：登录和权限还停留在原型阶段

问题：

当前“通过不同密码进入不同角色”的方式适合 MVP，但不适合作为长期商业系统。

风险：

- 密码泄露后无法追踪具体人。
- 店长调店、离职、权限变更很难管理。
- 老板角色预览和真实登录角色的边界需要后端强约束。

优化方案：

第一阶段不必做复杂 IAM，但要升级为：

- 用户账号
- 角色
- 绑定门店范围
- 是否启用
- 最近登录
- 密码重置
- 操作审计

验收标准：

- 每个操作能追溯到具体用户。
- 店长只能访问自己门店，后端强制。
- 禁用用户无法再登录。

## 5. 体验层面的问题

### 5.1 登录页

优点：

- 页面干净，视觉聚焦，橙色风格明确。

问题：

- 标题仍是“门店利润系统”，无法承载 AI Profit OS 的更大定位。
- 副标题写固定门店数量，容易过期。
- 提示“输入对应角色密码”对安全不友好。

建议：

- 标题改为“AI Profit OS”。
- 副标题改为“多门店经营异常处理系统”。
- 密码提示不要暴露角色体系。
- 登录后根据账号角色进入对应工作台。

### 5.2 今日待办

优点：

- 这是当前最重要、最正确的产品方向。
- 红 / 橙 / 蓝 / 灰优先级清晰。
- 一键跳转到处理页的设计正确。

问题：

- 统计数字和待办列表还没有体现 owner、截止时间、处理状态。
- 待办卡片上“为什么重要”的证据还可以更明确。
- 处理动作仍然依赖跳转后的页面是否接住上下文。

建议：

- 待办卡片增加：
  - 负责人
  - 截止时间
  - 来源数据
  - 当前处理状态
  - 是否已上报
- 红色风险列表增加“必须今天处理”区。

### 5.3 利润概览

优点：

- KPI、品牌卡、门店排名和趋势具备基本经营看板价值。
- 视觉风格统一，适合老板快速扫一眼。

问题：

- 仍偏结果展示，不够解释原因。
- 没有明确告诉老板“现在该让谁处理”。
- 亏损和低净利率门店缺少自动转待办。

建议：

- 增加“利润异常解释”。
- 对每个异常提供“查看明细”“生成待办”“上报/分派”。

### 5.4 报销栏

优点：

- 金额、状态、门店、类别在一个表里，基本可用。

问题：

- 审核状态过于简化。
- 缺少驳回原因和复核轨迹。
- 支付截图不是首要可见证据。

建议：

- 列表中显示凭证缩略图。
- 每条记录有时间线：提交、审核、驳回、完成。
- 大额报销自动进入老板待办。

### 5.5 仓库中心

优点：

- 已经能覆盖库存、叫货、入库、物料档案，业务面完整。

问题：

- 首屏任务过多。
- 日常处理和基础资料维护混在一起。
- 叫货单处理没有被置顶为核心任务。

建议：

- 先做“仓库处理台”，再做“仓库资料管理”。
- 叫货单列表置顶，按超时、待审核、待配货排序。
- 物料和采购表单折叠或放二级页。

## 6. 架构与可维护性风险

### 6.1 前端主文件过大

问题：

`index.html` 承担了样式、导航、页面结构、渲染、事件、业务计算、导入、巡店、仓库、工资等大量职责。继续堆功能会越来越难改。

影响：

- 改一个模块容易影响别的模块。
- 自动化测试难覆盖真实交互。
- 新人很难理解边界。

优化方案：

短期不重写，但要做“切片式拆分”：

- `todo-workbench.js`
- `expense-page.js`
- `warehouse-page.js`
- `inspection-page.js`
- `salary-page.js`
- `finance-report.js`
- `shared-ui.js`
- `api-client.js`

每拆一个模块，先保持功能不变，再接结构化 API。

### 6.2 兼容存储和结构化 API 并存

问题：

这是迁移期可以接受的状态，但不能长期存在。

优化方案：

- 给每个模块设定“数据归属状态”：
  - legacy
  - migrating
  - structured
  - readonly legacy
- 页面上不一定展示给普通用户，但产品和开发文档必须追踪。

### 6.3 测试覆盖仍偏技术层

问题：

已有 Controller / Service / 静态资源测试，但缺少真正的“角色主流程测试”。

建议增加端到端验收：

- 老板登录 -> 今日待办 -> 利润异常 -> 利润表。
- 财务登录 -> 今日待办 -> 待审核报销 -> 审核完成。
- 督导登录 -> 发起巡店 -> 生成整改 -> 店长看到整改。
- 店长登录 -> 只看到本店 -> 提交报销 / 叫货。
- 仓库登录 -> 处理叫货 -> 店长看到状态。

## 7. 优化路线图

### 7.1 未来 7 天：先修信任和闭环

目标：让系统从“能看”变成“能每天处理事”。

任务：

1. 统一产品名称、README、登录页和当前版本说明。
2. 完成角色待办后端接口第一版。
3. 每条待办增加 owner、dueAt、source、status。
4. 老板待办只展示利润异常和上报事项。
5. 财务、督导、仓库上报老板必须填写原因。
6. 建立数据健康页面或至少建立管理员可见的数据状态条。

### 7.2 未来 30 天：形成五个角色闭环

目标：老板、财务、督导、店长、仓库每天都能用。

任务：

1. 财务：报销审核、数据录入、工资核对闭环。
2. 督导：巡店记录、整改跟踪、超期提醒闭环。
3. 店长：本店整改、报销、叫货、经营提醒闭环。
4. 仓库：叫货处理、库存预警、配货出库闭环。
5. 老板：利润异常、上报事项、处理进度总览。

验收：

- 每个角色都有“今日待办 -> 处理页 -> 状态更新 -> 记录留痕”的路径。
- 老板可以看到所有红色风险的处理进度。

### 7.3 未来 60 天：把 AI 做成可信参谋

目标：AI 不只是聊天，而是基于证据的经营解释。

任务：

1. AI 总结支持证据引用。
2. AI 回答显示数据月份、门店范围和来源模块。
3. DeepSeek 失败时规则总结质量足够可用。
4. AI 不直接执行业务动作，只生成建议和解释。

验收：

- 老板能从 AI 总结点击到每条证据。
- 财务和店长不会看到超出权限的数据。

### 7.4 未来 90 天：准备商业化交付

目标：从自用系统变成可交付系统。

任务：

1. 多租户初始化。
2. 用户与权限管理。
3. 数据备份恢复。
4. 部署健康检查。
5. 手机端店长和督导关键流程。
6. 操作审计和异常告警。

验收：

- 新租户可以初始化并导入门店。
- 租户之间数据隔离有自动化测试。
- 线上服务能看到数据库、AI、YOLO、存储的健康状态。

## 8. 优先级任务表

| 优先级 | 问题 | 方案 | 验收标准 |
| --- | --- | --- | --- |
| P0 | 产品名和版本叙事不一致 | 统一 AI Profit OS 主线 | 登录页、README、部署文档一致 |
| P0 | 数据源不够可信 | 数据健康中心 + 结构化 API 优先 | 用户知道数据来源和更新时间 |
| P0 | 待办没有完整生命周期 | role_todo + escalation 状态模型 | 每条风险有负责人、状态、来源 |
| P0 | 店长权限必须后端锁定 | 店长接口按登录身份锁门店 | 前端传其他 storeId 无效 |
| P0 | 老板看不到处理进度 | 老板风险追踪页 | 红色风险全部有处理状态 |

## 9. 不建议现在做的事

1. 不建议现在全面重写 Vue 新前端。  
   当前最重要的是业务闭环和数据可信，重写会拖慢验证。

2. 不建议继续横向增加新模块。  
   运营中心、采购 V1、AI Agent 都可以做，但要等利润、报销、巡店、仓库的闭环先稳。

3. 不建议让 AI 自动处理业务。  
   现阶段 AI 应该解释、提醒、建议，不应该自动审批、自动出库、自动改数据。

4. 不建议把“今日待办”做成大而全操作页。  
   它应该是工作入口和追踪中心，复杂处理仍回到专业页面。

5. 不建议忽略移动端，但也不建议全站移动化。  
   先做店长和督导两个移动高频流程。

## 10. 理想产品形态

用户进入系统后，不应该想“我要去哪个菜单”，而应该看到：

- 老板：今天哪些店出问题，谁在处理，什么时候能处理完。
- 财务：哪些数据、报销、工资需要我核对。
- 督导：哪些整改超期，哪些门店要跟进。
- 店长：我的店今天要整改、报销、叫货、看经营。
- 仓库：哪些叫货单要处理，哪些物料有风险。

这才是 AI Profit OS 的价值。

## 11. 一句话结论

现在的项目已经有成为经营操作系统的基础，但上线前必须从“模块堆叠”转向“角色闭环”，从“能展示数据”转向“能让人处理问题”，从“AI 文案”转向“有证据的 AI 经营参谋”。
