# 门店利润系统业务流程试跑与产品化改造计划

日期：2026-07-07
范围：8080 旧版主线，`backend/src/main/resources/static/index.html` + Spring Boot/MySQL 后端
目标：把系统按真实企业使用方式跑一遍，找出跑不通、口径不稳、体验不够人性化的问题，并给出后续实现计划。

## 结论

当前系统已经不是一个空 Demo。管理员、老板、店长三类角色可以登录；利润概览、利润表、门店详情、督导巡店、门店助手、数据录入、报销、导出、门店管理、日志、平台入口、用户权限、员工工资等页面都能打开，自动化试跑没有发现前端 console error 或网络 4xx/5xx 异常。

但现在还不能按“可卖给企业”的标准交付。最核心的问题不是页面能不能打开，而是数据口径、权限口径和导入口径还不够稳：店长页面存在趋势数据越权混用；页面助手和直接后端助手会回答不同数字；导入预览可能把表格中的月份覆盖成当前页面月份；截图识别入口已经露出，但后端还只是占位。

## 项目结构判断

这个项目目前是“旧版单页应用 + 新后端逐步接管”的混合架构。

```text
index.html / database.js
  旧版单页入口、角色模拟、页面渲染、大量业务规则

backend/
  Spring Boot 3 后端
  托管静态页面
  提供 auth、finance、organization、assistant、imports、export 等接口

web/
  Vue 实验目录
  当前不是主线，本次没有触碰

docs/
  迁移方案、功能方案、审计文档
```

后端模块大致已经成形：

```text
assistant    AI 助手
finance      经营数据、利润数据
importing    截图/表格导入识别
organization 品牌、门店
platform     登录、用户、租户基础
storage      旧前端兼容存储
audit        操作日志
```

真实风险在于：旧前端仍掌握大量业务状态和本地规则，后端也有一套结构化数据。两套数据在部分场景会不一致，所以系统看起来能用，但一旦老板、店长、AI 助手分别从不同入口查同一个问题，就会出现“数字为什么不一样”的信任问题。

## 高优先级问题

### 1. 店长视角的数据权限不完整

店长 `rg1` 登录后，顶部指标只显示本店数据，这是对的；但“月度净利润趋势”仍使用了全部门店数据。页面截图里本店 2026-07 净利润为 69,867，但趋势显示 01 月 281 万、02 月 600 万这类集团级数据。

影响：店长看到不该看到的全局趋势，或者至少看到和本店不一致的数字。企业客户会直接质疑权限系统。

代码线索：

```text
backend/src/main/resources/static/index.html:1192
const pool=DASH_BRAND?STORES.filter(s=>s.brand===DASH_BRAND):STORES;
```

应改为基于 `visibleStores()` 计算趋势和品牌卡片，并且店长视角不展示其它品牌的 0/10、0/14 这类无意义卡片。

### 2. 助手前端回答和后端直接回答数字不一致

页面内问“保利店营业额”，页面助手基于前端可见上下文回答：

```text
2026-07，营业总收入 113,654，净利润 13,145
```

但直接调用 `/api/assistant/chat`，不带前端上下文时回答：

```text
2026-07，营业总收入 270,798，净利润 60,388
```

影响：同一个问题从不同入口得到不同答案。AI 助手最怕这个，因为用户不会区分“前端上下文”和“后端数据源”，只会认为系统在乱说。

建议：助手必须统一数据入口。短期要求所有助手请求都带 `contextEntries` 并在后端明确标记“本次使用前端可见数据”；中期将旧前端数据迁入统一 `finance` 数据源，前端只消费后端。

### 3. 表格导入月份存在被页面选择覆盖的风险

测试 CSV 中带了 `2099-12`，但预览显示为 `2026-07`。这说明当页面已经选定月份时，导入识别会优先采用页面默认月份，覆盖文件里的月份。

影响：财务人员导入多月表或错月表时，数据可能落到错误月份。这个问题比导入失败更危险，因为它看起来成功。

建议：导入预览必须把“文件识别月份”和“页面当前月份”分开显示；如果二者不一致，强制让用户确认，不能静默覆盖。

### 4. 截图识别入口已露出，但后端仍是占位

数据录入页已经有“截图识别”入口；后端截图类型识别接口返回成功壳，但实际提示视觉能力尚未配置。

影响：用户会以为已经可以拍照/截图自动识别，实际不能完成核心承诺。这个会伤害第一印象。

建议：如果短期不接 OCR/视觉模型，前端要明确标记“待配置”；如果要上线，就接入真实 OCR/视觉识别，并进入同一套导入预览、校验、提交流程。

### 5. 报销弹窗 Esc 无法关闭

新增报销弹窗可以打开，但按 Esc 不关闭，会挡住后续操作。用户可以点取消，但键盘逃生口缺失。

影响：这不是后端 bug，但会让系统显得不够顺手。企业后台里这类小问题会积累成“这个软件有点别扭”的感受。

建议：所有全局弹窗统一支持 Esc 关闭、点击遮罩关闭，并且关闭后恢复焦点。

## 中优先级问题

### 6. 成本缺失时显示 100% 利润率，容易误导

部分门店历史月份成本/费用为 0，页面直接算出 100% 净利率。真实业务里这通常不是“利润极好”，而是“成本未录入”。

建议：当收入存在但成本和费用同时为空时，状态显示为“数据不完整”，排名和 AI 分析中降低可信度，不参与优秀榜单。

### 7. 助手结果卡片金额换行不自然

助手回答里 `¥113,654` 在卡片里发生不自然换行，视觉上像系统没有打磨完。

建议：金额卡片设置最小宽度、禁止数字中间换行，移动端改为纵向排列。

### 8. 数据导出文案仍停留在本地/CloudBase 时代

导出页仍提示“本地存储模式、未连接云端、CloudBase”，但当前主线已经是 Spring Boot/MySQL。

建议：文案改成“当前连接本机/服务器数据库”，并区分“导出经营报表”和“备份迁移”。商业化版本不要出现过时技术词。

### 9. 用户权限页仍是本地模拟说明

页面说明“正式多用户登录需对接后端”，但后端已有 auth/session/users 雏形。现在处在半本地、半后端状态。

建议：短期把这页定位为“权限说明”；中期真正接入后端用户、角色、门店授权。

### 10. 平台登录只是外链，不是数据采集

平台登录入口能打开，但对老板来说这不是“接入平台”，只是“放了一排链接”。

建议：命名改成“平台入口”；真正的数据采集放到“数据导入中心/平台对接”中，展示连接状态、最近同步时间、失败原因。

## 人性化产品改造方向

### 老板视角

老板每天打开系统，不应该先看到复杂表格，而应该先看到三句话：

```text
今天/本月赚了多少钱
哪几家店拖后腿
下一步最该处理什么
```

建议首页结构：

```text
经营健康度
本月利润 / 利润率 / 环比
异常门店 3 个
AI 今日建议 1-3 条
```

老板可以下钻到表格，但第一屏要像经营驾驶舱，不像财务系统。

### 店长视角

店长只关心自己店。不要让店长看到 38 店、多个品牌、集团趋势这种信息。

建议店长首页：

```text
本店本月营业额
本店本月净利润
和上月比好还是差
今天需要补录什么
老板/督导给我的待办
```

店长的“门店助手”也应该自动锁定本店，不让他问到其它店的数据。

### 财务视角

财务最怕漏录、错录、重复录。

建议财务首页：

```text
本月 38 家，已录入 38 家
成本缺失 6 家
数据冲突 2 条
待审核报销 0 条
导入失败记录 0 条
```

数据导入中心应该比报表更重要，因为企业系统的生命线是“数据进得来、进得准”。

### 督导视角

督导巡店不是填表游戏，而是现场发现问题。

建议流程：

```text
选择门店
拍现场照片
AI 标出可能问题
督导确认扣分项
生成整改任务
下次巡检自动追踪
```

照片识别结果不要直接入库扣分，必须让人确认。AI 是辅助，不是最终裁判。

## 分阶段实现计划

### P0：先修可信度问题

目标：同一个人、同一个问题、同一个月份，系统只能有一个可信答案。

任务：

1. 修复店长视角趋势和品牌卡片权限过滤，所有统计统一从 `visibleStores()` 开始。
2. 统一助手数据口径，后端返回时明确 `dataSource=frontend-context/backend-finance`。
3. 导入预览增加月份冲突提示，文件月份和页面月份不一致时禁止一键提交。
4. 截图识别未配置时前端降级为明确的“待配置”，不伪装成可用。
5. 弹窗统一支持 Esc 关闭。

验收：

```text
店长 rg1 只能看到 rg1 的趋势和数据
保利店营业额在页面助手和接口助手口径一致
导入 2099-12 文件不会静默变成 2026-07
截图识别不可用时用户能明确知道原因
所有弹窗 Esc 可关闭
```

### P1：补齐数据录入闭环

目标：财务人员敢把真实 Excel 和截图交给系统。

任务：

1. 表格导入支持多 sheet、多门店、多月份。
2. 识别结果统一进入“预览-校验-冲突处理-提交”四步。
3. 冲突处理支持跳过、覆盖、另存为新版本。
4. 保存导入批次、原始文件名、操作者、时间、识别来源。
5. 导入后自动刷新利润概览和门店详情。

验收：

```text
同一份 Excel 可导入多家店
重复导入不会静默覆盖
导入失败能看到具体字段原因
导入成功后报表立刻更新
操作日志能追溯是谁导入的
```

### P2：改造角色首页

目标：让每个角色进来看到“自己的工作”，不是同一套复杂页面。

任务：

1. 老板首页：经营结论 + 异常门店 + AI 建议。
2. 店长首页：本店表现 + 补录提醒 + 整改任务。
3. 财务首页：录入完整度 + 冲突 + 报销待审。
4. 督导首页：待巡检门店 + 上次问题 + 拍照入口。
5. 全局搜索改成真正的“问助手/搜数据”入口。

验收：

```text
老板 10 秒内知道公司赚不赚钱
店长不会看到其它门店敏感数据
财务能一眼知道哪些数据没齐
督导能从手机完成拍照巡店
```

### P3：走向商业版架构

目标：从 38 店内用工具，升级为可卖给多企业的 AI 经营管理系统。

任务：

1. 多租户：tenant、user、role、store、permission 全面落库。
2. RBAC + 数据权限：后端强制校验，不只靠前端隐藏。
3. 经营数据统一：前端不再维护独立 ENTRIES 作为主数据。
4. AI Gateway：DeepSeek/GPT/Claude 不直接写在业务逻辑里。
5. Prompt 模板管理：行业、角色、场景可配置。
6. 审计日志：导入、修改、删除、AI 分析都可追溯。

验收：

```text
不同企业数据物理/逻辑隔离
店长接口层也无法访问其它店
AI 回答每个数字都能回溯到数据源
换模型不影响业务代码
```

## 建议下一步

建议下一轮不要再扩新功能，先做 P0。顺序如下：

1. 修店长权限统计。
2. 修助手统一数据口径。
3. 修导入月份冲突。
4. 修截图识别状态文案。
5. 修弹窗 Esc。

这五个修完，系统的“可信度”会明显上一个台阶。之后再继续做截图 OCR、Excel 多表导入、角色首页，商业化基础会更稳。
