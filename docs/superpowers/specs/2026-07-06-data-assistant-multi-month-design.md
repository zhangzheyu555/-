# 数据助手多月份查询改造设计

日期：2026-07-06

## 背景

当前数据助手已改为本地简单问答，不接 AI。它可以识别门店、品牌、指标、人员和月份，但默认月份被写死为 `2026-07`。

现有种子经营数据主要覆盖 `2026-01` 到 `2026-05`。因此如果用户问“保利店营业额”或点快捷按钮“7月净利排名”，系统会优先查 7 月，而 7 月暂无数据，体验上像是“其他月份查不到”。

本次改造目标是：保留简单问答，不接 AI，让数据助手能自然查询已有任意月份，并清楚提示哪些月份有数据。

## 目标

- 用户明确输入月份时，严格按该月份查询。
- 用户输入“各月 / 每月 / 趋势 / 全部月份”时，返回该门店或品牌的多月份表格。
- 用户输入“1-5月 / 1到5月 / 2026年1月至5月”时，按月份范围查询。
- 用户没有输入月份时，默认查询“最新已录入月份”，而不是硬编码 7 月。
- 如果用户明确问 7 月但暂无数据，要直接提示“7 月暂无经营数据”，并给出已有月份快捷入口。
- 数据助手仍然只是简单规则查询，不接 AI，不做自由文本智能分析。

## 非目标

本次不做后端数据助手接口，不做 AI 问答，不重写 UI，不迁移利润计算到 Java 后端。

这次只调整前端数据助手的本地规则引擎。

## 推荐方案

采用“本地规则引擎增强”的方案。

理由：

- 当前前端已经有 `ENTRIES`、`STORES`、`SALARY` 和 `calc()`，能直接生成结果。
- 改动范围集中在 `index.html` 数据助手区域。
- 不影响刚搭好的 Java 后端第一阶段。
- 后续迁移到后端时，可以把这些规则搬到 `/api/data-assistant/simple-query`。

备选方案：

- 方案 A：继续硬编码当前月份。实现最少，但 7 月没数据时体验差，不推荐。
- 方案 B：直接做后端数据助手 API。架构更长远，但现在后端还处于兼容存储阶段，投入偏大。
- 方案 C：先增强前端规则引擎。最适合当前阶段，推荐。

## 查询语义设计

### 月份优先级

查询月份按以下顺序确定：

1. 用户明确输入的月份。
2. 用户输入的月份范围。
3. 用户输入“各月 / 每月 / 趋势 / 全部月份”等多月份意图。
4. 没有输入月份时，使用当前数据范围内的最新已录入月份。
5. 如果没有任何经营数据，则使用当前自然月份作为兜底，并提示暂无数据。

当前数据中，默认最新已录入月份应为 `2026-05`。

### 支持的月份表达

单月：

- `1月`
- `一月`
- `2026-01`
- `2026年1月`
- `2026/01`

范围：

- `1-5月`
- `1到5月`
- `1月至5月`
- `2026年1月至5月`
- `2026-01到2026-05`

多月：

- `各月`
- `每月`
- `趋势`
- `全部月份`
- `月度`

### 默认月份行为

旧行为：

```text
保利店营业额 -> 查 2026-07
各店净利排名 -> 查 2026-07
```

新行为：

```text
保利店营业额 -> 查最新已录入月份，当前为 2026-05
各店净利排名 -> 查最新已录入月份，当前为 2026-05
```

如果用户明确问：

```text
保利店7月营业额
```

则仍查 `2026-07`，并在无数据时提示：

```text
2026年7月暂无经营数据。当前已有经营数据月份：2026年1月、2月、3月、4月、5月。
```

## 回答类型设计

### 单店单月指标

示例：

```text
保利店3月营业额
```

返回保利店 2026 年 3 月经营卡片，包括：

- 营业总收入
- 实收收入
- 净利润
- 净利率
- 成本合计
- 费用合计

### 单店多月趋势

示例：

```text
保利店各月营业额
保利店1-5月净利
```

返回一个表格：

| 月份 | 营业额 | 实收 | 净利润 | 净利率 |
| --- | --- | --- | --- | --- |

如果用户指定了指标，例如“营业额”，该指标要高亮展示。

### 全部门店单月排名

示例：

```text
3月各店净利排名
4月哪些店亏损
```

返回指定月份的门店排名或亏损筛查。

### 全部门店多月汇总排名

示例：

```text
1-5月各店净利排名
1到5月营业额最高的店
```

返回按门店聚合的区间排名。

聚合规则：

- 金额类指标：按月份求和。
- 净利率：用区间总净利润 / 区间总营业收入计算。
- 亏损筛查：默认筛选区间总净利润小于 0 的门店。

### 品牌各月汇总

示例：

```text
霸王茶姬各月营收
瑞幸咖啡每月净利
```

返回品牌在每个月的汇总表。

当前已有 `botBrandMonthly()`，保留并增强指标展示。

### 人员和工资查询

人员查询逻辑保持现状：

- 明确月份时，查该月工资。
- 没有月份时，查工资数据中最新月份。

示例：

```text
荆州之星5月人员名单
美佳华店员工工资
```

## 需要新增或调整的函数

### 月份辅助函数

新增：

- `botAllEntryMonths()`：返回所有经营数据月份。
- `botLatestEntryMonth()`：返回最新已录入经营月份。
- `botAvailableMonthsForStores(stores)`：返回指定门店集合已有月份。
- `botMonthHint(months)`：生成“当前已有数据月份”提示。
- `botResolveMonthIntent(q, stores, brand)`：统一解析单月、范围、多月和默认月份。

调整：

- `botParseMonths(q)`：继续负责单月解析，但补充范围解析。
- `mLabel(m)`：保持现状。

### 回答函数

新增：

- `botStoreMultiMonth(stores, months, metric)`：单店或多店的多月份经营表。
- `botRangeRank(q, months, metric)`：多月份区间排名。
- `botNoMonthData(m, scopeMonths)`：无数据提示，附可点击月份建议。

调整：

- `botRank(q, m, mt)`：无数据时显示已有月份建议。
- `botStoreCard(s, m, mt)`：无数据时显示该店已有月份。
- `botKeyword(q)`：点击门店后不再固定跳 7 月，改为最新已录入月份或“各月经营”。
- `localAnswer(q)`：使用统一月份意图对象，不再直接用 `months[0] || BOT_DEFAULT_MONTH`。

### 快捷问题

旧快捷问题偏 7 月：

- 保利店7月营业额
- 7月各店净利排名
- 7月哪些店在亏损

新快捷问题建议：

- 保利店5月营业额（最新已录入月份动态生成）
- 3月各店净利排名
- 保利店各月营业额
- 霸王茶姬各月营收
- 1-5月各店净利排名
- 荆州之星店人员名单

快捷文案里的最新月份要从数据动态读取，不再写死。

## 数据流

```text
用户输入
  -> botAsk
  -> localAnswer
  -> 识别门店/品牌/人员/指标
  -> botResolveMonthIntent
  -> 按查询类型分发
     -> 单店单月
     -> 单店多月
     -> 单月排名
     -> 区间排名
     -> 品牌各月
     -> 人员工资
  -> 生成 HTML 表格或提示
```

## 边界处理

### 问未来月份

示例：

```text
保利店7月营业额
```

如果没有 7 月数据，直接提示无数据，并展示已有月份入口。

### 问不存在门店

保持当前简单提示，引导用户输入门店名、月份、指标。

### 问多个门店

如果用户输入多个门店名，单月查询返回多个门店卡片；多月查询返回多个门店的分组表格。

### 问范围但部分月份无数据

只聚合有数据的月份，并在结果顶部提示：

```text
已按有数据月份统计：2026年1月、2月、3月、4月、5月。
```

### 店长角色

继续使用 `visibleStores()` 和 `isVisibleStore()`，只查询当前角色可见门店。

- 明确月份 1-5 月均可查。
- 明确 7 月但无数据时，提示清楚，不误导。
- 不输入月份时，默认使用最新已录入月份。
- “各月 / 趋势 / 1-5月”能返回多月份表格或区间排名。
- 数据助手仍不调用 AI，也不请求外部网络。
- `index.html` 脚本语法检查通过。
