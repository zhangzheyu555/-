# MySQL 数据归属与虚拟数据边界

日期：2026-07-08  
适用产品：AI Profit OS / 多门店经营异常处理系统  
适用范围：利润、报销、巡店、仓库、工资、今日待办等真实业务数据

## 1. 总原则

真实业务数据必须写入 MySQL。浏览器、本地缓存、兼容 KV、前端静态种子数据，只能作为迁移期读取来源或演示数据来源，不能作为长期真实数据源。

虚拟数据不得入库。用于演示、占位、前端样式验证、mock 接口验证的数据，不能写入正式 MySQL 业务表，不能混入真实经营报表。

## 2. 数据分类

| 类型 | 能否入 MySQL | 说明 |
| --- | --- | --- |
| 真实经营数据 | 必须入库 | 门店实际收入、成本、费用、利润、工资、报销、巡店、仓库流水。 |
| 迁移期旧数据 | 迁移后入库 | 从浏览器旧数据或兼容 KV 读取后，需要经过迁移校验再进入 MySQL。 |
| 示例数据 | 不得入库 | 用于产品演示、开发样式、浏览器 mock 的数据。 |
| 前端静态种子数据 | 不得作为长期真实源 | 可以用于冷启动展示，但不能替代 MySQL 真实表。 |
| AI 总结文本 | 可记录，但不能替代事实表 | AI 结论必须能追溯到真实 MySQL 数据来源。 |

## 3. 关键模块归属

| 模块 | 当前前端状态 | 最终 MySQL 归属 | 说明 |
| --- | --- | --- | --- |
| 今日待办 | MySQL结构化 | 角色待办 / 异常处理表 | 由后端分角色接口生成，不允许前端模糊判断。 |
| 利润 | 兼容KV | 月度利润表、门店经营明细表 | 真实收入、成本、费用、净利率必须结构化。 |
| 报销 | 兼容KV | 报销单、凭证、审核流水表 | 必须记录审核人、审核时间、驳回原因、入账月份。 |
| 巡店 | 兼容KV | 巡店记录、整改项、照片/识别绑定表 | 必须能追溯门店、督导、整改状态、复查结果。 |
| 仓库 | 兼容KV | 物料、库存、叫货、采购、出入库流水表 | 真实库存变化必须来自流水，不能只存最终数。 |
| 工资 | 兼容KV | 员工、出勤、工资项、提成、审核表 | 真实工资必须可审计，测算样例不能入库。 |

## 4. 浏览器数据使用边界

允许：

- 读取迁移前遗留的浏览器旧数据。
- 在本地开发或无后端场景下展示演示数据。
- 用 mock 数据验证前端布局和交互。

不允许：

- 将真实经营数据长期保存在 `localStorage`、`sessionStorage` 或前端静态文件中。
- 将浏览器数据当作老板、财务、店长看到的正式报表来源。
- 将 mock、seed、demo 数据写入正式 MySQL 业务表。
- 让 AI 总结基于无法追溯来源的浏览器数据输出正式经营建议。

## 5. 迁移验收标准

1. 老板进入数据健康页，可以看到每个模块的数据来源状态。
2. 真实业务写入必须经过后端接口进入 MySQL。
3. 浏览器旧数据只允许作为迁移输入，迁移后需要标记来源和迁移时间。
4. 每个报表、待办、AI 总结都能显示数据来源和更新时间。
5. 示例数据、mock 数据、静态种子数据不得进入正式业务表。
6. 店长、财务、督导、仓库管理员不能看到老板的数据健康入口。

## 6. 后续落地建议

第一阶段先建立数据健康可视化和文档边界。  
第二阶段逐模块把真实写入改到 MySQL 结构化表。  
第三阶段关闭浏览器真实数据写入，只保留兼容读取和迁移工具。  
第四阶段给每个模块增加数据来源、最后更新时间、迁移状态和异常提示。
