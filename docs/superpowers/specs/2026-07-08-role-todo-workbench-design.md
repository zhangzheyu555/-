# 多角色今日待办工作台设计

**日期：** 2026-07-08  
**项目：** AI Profit OS / AI 企业经营分析系统  
**设计目标：** 在不大改现有单页系统的前提下，新增“今日待办”角色首页，让不同角色登录后直接看到自己该处理的事项，并能一键跳转到现有业务页面完成处理。

## 1. 核心结论

本次前端改造不推翻现有结构：保留当前左侧菜单、顶部栏、现有功能页和橙色品牌风格，只新增一个左侧入口“今日待办”，并把它作为登录后的默认首页。

系统通过登录密码识别角色和门店范围：

- 店长：只能看到自己门店的数据和待办。
- 老板：可切换品牌、门店，可看汇总。
- 财务：可切换品牌、门店，处理财务相关待办。
- 督导：可切换品牌、门店，优先处理待整改事项。
- 仓库管理员：可切换品牌、门店，优先处理门店叫货单。
- 运营：第一版只保留占位，不作为主功能建设。

待办判断必须来自后端标准接口，前端不能自行模糊计算异常。前端只负责展示、筛选、跳转和状态呈现。

## 2. 设计原则

1. 人先于功能  
   用户进入系统时，不应该先理解菜单，而应该直接知道“我今天要处理什么”。

2. 不做大改  
   保留现有左侧菜单和顶部结构，现有功能页继续作为处理页面。

3. 每个角色只看自己的工作  
   店长不看到别的店；老板、财务、督导、仓库管理员可以按权限切换品牌和门店。

4. 待办只做导航，不直接处理业务  
   今日待办页不做“审核通过”“整改完成”“出库完成”等业务动作。真正处理必须跳到对应现有功能页。

5. 后端标准化判断  
   所有异常、待处理、提醒、已处理状态都由后端角色接口返回。前端不根据本地数据猜测。

6. 已处理默认隐藏  
   灰色“已处理”默认不显示，只在用户打开筛选后可查看。

## 3. 角色首页优先级

### 3.1 老板

老板首页是“AI 经营参谋”，不是普通报表入口。

第一屏优先展示：

- DeepSeek 生成的自然语言经营重点。
- DeepSeek 不可用时，显示后端规则兜底总结。
- 利润异常待办。
- 其他角色上报给老板的事项。

老板的利润异常第一版包含三类：

- 门店净利润为负。
- 门店净利率低于 20%。
- 门店收入环比明显下降，默认阈值为较上月下降超过 20%。

建议严重程度：

- 净利润为负：红色风险。
- 收入环比下降超过 30%：红色风险。
- 收入环比下降 20% 到 30%：橙色待处理。
- 净利率低于 20% 但净利润仍为正：橙色待处理。

### 3.2 财务

财务首页是并列待办中心，不强行只突出一个任务。

第一屏平级展示：

- 待录入 / 待核对数据。
- 报销审核与费用异常。
- 工资相关事项。
- 利润表 / 数据导出事项。

财务处理不了的事项可以上报老板，必须填写原因。

### 3.3 督导

督导首页第一优先级是待整改跟踪。

第一屏展示：

- 红线问题未整改。
- 巡店扣分项待跟进。
- 即将超期或已经超期的整改。
- 发起新巡店入口和历史巡店分析作为次级入口。

督导处理不了的整改事项可以上报老板，必须填写原因。

### 3.4 店长

店长首页必须只看自己的门店。

优先级：

1. 督导巡查结果和待整改提醒。
2. 叫货 / 仓库申请。
3. 提交报销。
4. 本店经营数据辅助查看，不抢第一屏主位置。

店长点击待办后跳到自己的门店上下文，不允许切到其他门店。

### 3.5 仓库管理员

仓库首页第一优先级是处理门店叫货单。

第一屏展示：

- 待处理叫货单。
- 缺货或安全库存预警。
- 采购 / 入库记录入口。

仓库处理不了的叫货事项可以上报老板，必须填写原因。

## 5. 页面布局设计

### 5.1 桌面端整体布局

保持当前橙色品牌风格和现有卡片语言。第一版重点保证电脑端，手机端只保证不崩，后续单独优化。

│ ...           │ 老板：AI 今日经营重点                         │

│               │ └──────────────────────────────────────────┘ │

### 5.2 顶部区域

每个角色显示不同标题：

- 老板：今天建议你重点看这些经营问题
- 财务：今天需要你核对和处理的财务事项
- 督导：这些门店整改需要跟进
- 店长：你门店今天需要处理的事项
- 仓库管理员：门店叫货和库存处理

右侧筛选：

- 品牌筛选：老板、财务、督导、仓库管理员可用。
- 门店筛选：老板、财务、督导、仓库管理员可用。
- 店长：品牌和门店锁定为自己门店，只展示不可编辑标签。
- 状态筛选：默认显示红色风险、橙色待处理、蓝色提醒；灰色已处理默认隐藏。
- 刷新按钮：重新请求当前角色接口。

### 5.3 统计条

统计条使用四档状态：

- 红色风险：需要尽快处理或上报。
- 橙色待处理：正常待处理。
- 蓝色提醒：需要关注，但不一定立即处理。
- 灰色已处理：默认隐藏，筛选打开后显示。

统计条示例：

```text
红色风险 2    橙色待处理 8    蓝色提醒 3    已处理 12
```

已处理默认不参与主列表展示。

### 5.4 待办卡片

每条待办都要包含：

- 优先级颜色。
- 角色可理解的标题。
- 门店 / 品牌 / 月份或业务日期。
- 一句话说明为什么重要。
- 来源模块。
- 更新时间或截止时间。
- 主按钮：进入处理页面。

示例：

```text
红色风险
霸王茶姬 · 保利店  2026-06 净利润为负
本月净利润 -3,200，净利率 -4.8%，建议先查看收入和成本明细。
来源：利润数据 · 更新时间：2026-07-08 09:30
[进入利润表]
```

### 5.5 空状态

空状态不能只写“暂无数据”，要告诉用户当前没事要处理。

示例：

- 老板：当前没有未处理利润风险。
- 财务：当前没有待核对财务事项。
- 督导：当前没有未完成整改。
- 店长：你的门店当前没有待处理事项。
- 仓库管理员：当前没有待处理叫货单。

## 6. 后端接口设计

不做统一 `/api/todos`。按角色拆接口。

### 6.1 角色待办接口

```http
GET /api/boss/todos
GET /api/finance/todos
GET /api/supervisor/todos
GET /api/store-manager/todos
GET /api/warehouse/todos
GET /api/operations/todos
```

通用查询参数：

```text
brandId       可选；店长忽略，由后端锁定
storeId       可选；店长忽略，由后端锁定
status        可选；RISK / PENDING / REMINDER / DONE
includeDone   可选；默认 false
limit         可选；默认 50
```

店长接口必须以后端登录身份锁定门店，不信任前端传入的 `storeId`。

## 7. 老板 DeepSeek 总结

### 7.1 调用原则

老板接口 `/api/boss/todos` 返回 `aiSummary`。

后端先生成结构化经营证据包，再调用 DeepSeek：

- 利润为负门店。
- 净利率低于 20% 门店。
- 收入环比下降超过 20% 门店。
- 已上报给老板但未处理的事项。

DeepSeek 只生成自然语言总结，不负责判定异常。异常判定必须由后端规则完成。

### 7.2 兜底策略

如果 DeepSeek 不可用：

- `aiSummary.source = "RULE_FALLBACK"`
- 使用后端规则生成总结。
- 页面继续展示，不出现空白。

示例：

```json
{
  "text": "今天建议先看 2 个利润风险：保利店净利润为负，桂东店收入环比下降 26%。",
  "source": "RULE_FALLBACK",
  "fallbackReason": "DEEPSEEK_TIMEOUT"
}
```

## 8. 上报老板机制

第一版支持三类来源上报老板：

- 财务报销 / 费用。
- 督导整改。
- 仓库叫货。

上报必须填写原因，不能一键空上报。

### 8.1 角色内上报接口

不使用统一上报入口，按来源角色拆分：

```http
POST /api/finance/todos/{todoId}/escalate
POST /api/supervisor/todos/{todoId}/escalate
POST /api/warehouse/todos/{todoId}/escalate
```

请求体：

```json
{
  "reason": "该笔费用缺少凭证，门店无法补齐，需要老板确认是否放行。",
  "severity": "RISK"
}
```

响应：

```json
{
  "success": true,
  "message": "已上报老板",
  "data": {
    "escalationId": "esc-20260708-0001",
    "bossTodoId": "boss-escalation-esc-20260708-0001"
  }
}
```

老板接口中展示未处理上报事项，点击后跳回来源业务页。

## 9. 前端跳转协议

待办点击后跳到现有功能页，并带上下文参数。

前端需要实现一个轻量路由函数：

```text
openTodoAction(action)
```

支持目标：

- `report`：利润表。
- `detail`：门店详情。
- `inspect`：督导巡店。
- `entry`：数据录入。
- `expense`：报销栏。
- `warehouse`：仓库中心。
- `salary`：员工工资。
- `export`：数据导出。

示例跳转：

```json
{
  "targetTab": "inspect",
  "params": {
    "storeId": "bw1",
    "inspectionId": "insp-1001",
    "mode": "detail"
  }
}
```

前端行为：

1. 切换到 `inspect` 标签。
2. 设置当前门店为 `bw1`。
3. 打开对应巡检详情。
4. 如果目标数据不存在，显示温和错误提示，不静默失败。

## 10. 角色接口第一版业务规则

### 10.1 `/api/boss/todos`

返回：

- 利润为负。
- 净利率低于 20%。
- 收入环比下降超过 20%。
- 其他角色上报老板且未处理的事项。

不直接展示普通报销、普通整改、普通叫货，这些由对应角色先处理。

### 10.2 `/api/finance/todos`

返回：

- 待录入或待核对利润数据。
- 待审核报销。
- 费用异常。
- 工资待核对。
- 导出提醒。

财务类待办重要性并列，通过状态颜色和时间排序。

### 10.3 `/api/supervisor/todos`

返回：

- 红线问题未整改。
- 巡检扣分项未整改。
- 整改超期。
- 即将到期整改。

默认规则：

- 超过整改截止时间：红色风险。
- 距截止时间 2 天内：橙色待处理。
- 新巡检结果通知：蓝色提醒。

### 10.4 `/api/store-manager/todos`

返回当前店长所属门店的事项：

- 待整改巡检结果。
- 叫货 / 仓库申请相关事项。
- 报销提交或退回提醒。
- 本店经营数据提醒。

店长接口必须根据后端用户身份确定 `storeId`。

### 10.5 `/api/warehouse/todos`

返回：

- 待处理门店叫货单。
- 已超时叫货单。
- 库存不足提醒。
- 采购 / 入库待确认。

优先级：

- 超时未处理叫货单：红色风险。
- 新叫货单：橙色待处理。
- 库存接近安全线：蓝色提醒。

### 10.6 `/api/operations/todos`

第一版返回占位：

- stats 全部为 0。
- items 为空。
- 文案提示“运营工作台后续开放”。

## 11. 错误与加载状态

### 11.1 加载中

页面显示当前角色文案：

```text
正在读取你的待办事项...
```

### 11.2 接口失败

不能白屏。显示：

```text
今日待办加载失败，请稍后重试。你仍然可以通过左侧菜单进入各业务页面。
```

提供按钮：

- 重新加载。
- 返回利润概览。

### 11.3 DeepSeek 失败

不显示错误给老板，直接展示规则兜底总结，并在角落用小字标识：

```text
规则总结
```

## 13. 分阶段实施计划

### Phase 1：接口契约和后端角色待办骨架

- 新增 6 个角色接口。
- 返回统一响应结构。
- 店长接口按后端身份锁定门店。
- operations 返回占位。
- 不接前端前先写接口测试。

### Phase 2：老板利润异常与 DeepSeek 总结

- 后端实现利润异常规则。
- 接入 DeepSeek 生成老板总结。
- DeepSeek 失败时返回规则兜底总结。
- 老板接口包含利润异常和上报事项。

### Phase 3：财务 / 督导 / 店长 / 仓库待办

- 财务：待录入、待审核、工资、导出。
- 督导：待整改、超期、即将到期。
- 店长：本店整改、叫货、报销、本店经营提醒。
- 仓库：叫货单、库存预警、采购入库。

### Phase 4：前端新增“今日待办”

- 左侧菜单新增入口。
- 登录后默认进入今日待办。
- 保持当前橙色风格。
- 渲染顶部统计、筛选器、待办列表。
- 已处理默认隐藏。

### Phase 5：一键跳转现有功能页

- 实现 `openTodoAction(action)`。
- 支持跳转利润表、巡店、仓库、报销、工资、导出等现有页。
- 带上品牌、门店、月份、单据、巡检上下文。

### Phase 6：上报老板

- 财务、督导、仓库业务页增加上报入口。
- 上报必须填写原因。
- 老板今日待办显示上报事项。

## 14. 验收标准

1. 不登录不能访问角色待办数据。
2. 店长登录后只能看到自己门店待办。
3. 老板、财务、督导、仓库管理员可以按品牌和门店筛选。
4. 已处理事项默认隐藏。
5. 每条待办都能跳到现有处理页面。
6. 今日待办页不直接处理业务动作。
7. 待办异常判断全部来自后端接口。
8. 老板首页优先显示 DeepSeek 总结；DeepSeek 不可用时显示规则兜底总结。
9. 财务、督导、仓库上报老板必须填写原因。
10. 保留现有左侧菜单和顶部结构，不做整站重构。

## 15. 非目标

第一版不做：

- 手机端深度优化。
- 全新前端架构重写。
- 统一 `/api/todos` 大接口。
- 前端自行计算异常。
- 在今日待办页直接完成审核、整改、出库等业务处理。
- 运营角色完整工作台。

## 16. 需要后续确认的配置项

以下默认值可先落地，后续做成系统配置：

- 低净利率阈值：20%。
- 收入环比明显下降阈值：20%。
- 收入环比严重下降阈值：30%。
- 督导整改临期提醒：截止前 2 天。
- 叫货单超时阈值：超过后端设定处理时限。

## 17. 一句话产品定位

“今日待办”不是新的报表页，而是每个角色进入系统后的工作入口：系统先告诉人该处理什么，人再一键进入对应页面处理。
