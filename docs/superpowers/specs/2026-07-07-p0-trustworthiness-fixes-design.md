# P0 可信度修复设计

日期：2026-07-07

## 背景

当前系统已经可以作为真实本地应用运行：Spring Boot 托管 8080 主线页面，后端提供认证、门店、财务、导入、仓库、助手等接口，老板、店长、财务、督导、仓库管理员等角色也已经有入口。

但业务试跑发现，系统的主要风险不是页面打不开，而是“同一份业务事实在不同入口看起来不一致”。例如店长视角部分趋势仍混入集团数据，页面助手和后端助手可能回答不同金额，导入文件里的月份可能被当前页面月份覆盖。对于经营系统来说，这类问题会直接破坏用户信任。

本阶段目标不是继续扩功能，也不是启动大重构，而是先把当前 8080 主线的数字口径、权限口径、导入口径和基础交互修稳。

## 目标

- 店长只看到自己门店相关的统计、趋势、排名和助手上下文。
- AI 助手回答必须说明数据来源，避免同一问题从不同入口得到无法解释的数字。
- 导入预览必须显式处理文件月份和页面月份冲突，不能静默改写月份。
- 截图识别未配置时明确降级，不伪装成已经可用。
- 主要弹窗支持 Esc 关闭，降低操作阻塞感。
- 保持当前 8080 主线、Spring Boot 后端、MySQL 数据、仓库 MVP 和现有角色入口继续可用。

## 非目标

- 不做 Vue `web/` 实验目录改造。
- 不做全面前端拆分。
- 不做完整 SaaS 多租户改造。
- 不接入新的 OCR 或视觉模型供应商。
- 不把 DeepSeek Key、数据库密码或任何密钥写入代码、文档、HTML 或 Git 历史。
- 不重写导入中心，只修正当前导入预览的可信度问题。

## 方案比较

### 方案一：只修前端表现

主要修改 `index.html` 和 `database.js`。优点是快，能解决店长趋势、导入月份提示、截图识别文案和弹窗 Esc。缺点是助手数据口径仍可能和后端直接接口存在解释差异。

### 方案二：前端修口径，后端补数据来源标记

保留当前 8080 主线，前端统一使用可见数据范围，后端助手返回 `dataSource`、`month`、`storeScope` 等元信息。优点是改动可控，又能让数字“可解释、可追溯”。这是本阶段推荐方案。

### 方案三：直接进入后端主数据源重构

长期最干净，但会牵动旧页面主数据、导入、助手、权限和报表，短期风险过高，不适合作为 P0 修复。

## 推荐方案

采用方案二：前端修口径，后端补数据来源标记。

本阶段围绕五个小闭环推进：

1. 店长权限统计。
2. 助手数据口径。
3. 导入月份冲突。
4. 截图识别状态。
5. 弹窗 Esc。

## 设计细节

### 1. 店长权限统计

问题：店长登录后，顶部指标已经趋向本店数据，但部分趋势和品牌卡片仍使用全量 `STORES` 或全局 `ENTRIES`。这会导致店长看到集团级趋势或其它品牌的空卡片。

设计：

- 所有店长视角统计从 `visibleStores()` 开始取门店范围。
- 月度趋势、品牌卡片、排名表、门店详情入口、助手上下文都基于可见门店集合。
- 店长角色下不展示其它品牌的空品牌卡片。
- 如果店长只有一个门店，默认锁定该门店，不提供跨店筛选。
- 老板、财务、督导、仓库管理员仍按各自角色保留现有可见范围。

验收：

- 店长 `rg1` 登录后，趋势图只包含 `rg1` 数据。
- 店长 `rg1` 不看到其它品牌的 0/10、0/14 空卡片。
- 店长无法从排名或助手上下文中拿到其它门店数据。

### 2. 助手数据口径

问题：页面助手带前端可见上下文时，可能回答一套数字；直接调后端 `/api/assistant/chat` 且不带上下文时，可能使用另一套数据源并回答不同数字。

设计：

- 前端页面助手继续传入当前用户可见范围内的结构化上下文。
- 后端响应增加元信息：
  - `dataSource`: `FRONTEND_CONTEXT`、`BACKEND_FINANCE`、`LOCAL_RULES`、`AI_ENRICHED`。
  - `month`: 本次回答使用的月份。
  - `storeScope`: `ALL_VISIBLE_STORES` 或具体门店 ID 列表。
  - `warnings`: 数据不完整、无上下文、未启用 AI 等提示。
- 如果请求带 `dataContext`，后端优先基于前端上下文回答，并在返回中标明 `FRONTEND_CONTEXT`。
- 如果请求不带 `dataContext`，后端只允许基于当前登录用户权限范围内的 finance 数据回答，并标明 `BACKEND_FINANCE`。
- 如果后端缺少足够数据，回答应说明“当前数据源不足”，而不是猜测。
- DeepSeek 只用于润色和分析，不作为未经约束的事实来源。

验收：

- 页面助手和后端助手回答同一门店同一月份时，要么数字一致，要么清楚标明数据源差异。
- 返回结构中能看到 `dataSource`。
- 无权限门店不会出现在助手上下文或后端查询结果中。

### 3. 导入月份冲突

问题：表格导入时，如果文件中包含月份，但页面当前也选了月份，当前页面月份可能覆盖文件月份。这比导入失败更危险，因为用户会以为导入成功。

设计：

- 导入识别结果保留两个月份字段：
  - `sourceMonth`: 文件或识别内容中的月份。
  - `pageMonth`: 页面当前选择月份。
- 如果 `sourceMonth` 和 `pageMonth` 一致，可以正常预览。
- 如果二者不一致，预览行标记为 `MONTH_CONFLICT`。
- 月份冲突行不能一键保存；用户必须选择：
  - 使用文件月份。
  - 使用页面月份。
  - 跳过该行。
- 批量导入时，每一行独立显示冲突状态。
- 手工录入表单仍使用页面当前月份，不受导入冲突逻辑影响。

验收：

- 上传包含 `2099-12` 的文件时，不会静默变成 `2026-07`。
- 冲突行显示文件月份和页面月份。
- 冲突未确认前，保存按钮不可直接提交该行。

### 4. 截图识别状态

问题：截图识别入口已经露出，但后端视觉能力可能未配置。用户点击后如果只看到成功壳或模糊失败，会以为产品承诺落空。

设计：

- 前端在截图识别区域显示当前能力状态：
  - 已配置：可以生成识别预览。
  - 未配置：显示“截图识别待配置，可先使用 Excel/CSV 或手工录入”。
- 后端截图识别不可用时返回明确错误码，例如 `VISION_NOT_CONFIGURED`。
- 前端收到该错误码时进入明确降级状态，不弹出技术错误。
- 未配置时，截图入口可以保留，但按钮文案应避免暗示已经可识别。

验收：

- 未配置视觉能力时，用户能看懂为什么不能截图识别。
- 前端没有未捕获异常。
- Excel/CSV 和手工录入入口仍可用。

### 5. 弹窗 Esc

问题：报销、工资、巡店、导入等弹窗如果无法用 Esc 关闭，会造成阻塞感。

设计：

- 建立一个轻量的全局 Esc 关闭机制。
- 当前打开的最上层弹窗响应 Esc。
- 点击遮罩关闭适用于安全弹窗；删除确认或会丢失编辑内容的弹窗需要保留确认提示。
- 关闭后清理错误提示和临时状态。
- 关闭后尽量恢复焦点到触发按钮或当前模块主区域。

验收：

- 报销新增弹窗按 Esc 可关闭。
- 工资新增弹窗按 Esc 可关闭。
- 巡店新增弹窗按 Esc 可关闭。
- 导入预览弹窗或面板关闭后不会留下遮罩或阻塞页面。

## 数据与安全

- 本阶段不写入任何密钥。
- 不新增敏感配置文件。
- 不提交 `output/`、截图、临时验证文件。
- 写接口继续走后端权限与审计。
- 导入冲突修复不自动覆盖正式数据。
- 助手数据源标记不暴露内部密钥、数据库连接或系统提示词。

## 测试计划

### 接口验证

- `GET /api/health` 返回 200。
- `POST /api/auth/login` 老板账号可登录。
- `GET /api/session/me` 返回当前角色和可见范围。
- `POST /api/assistant/chat` 返回 `dataSource` 元信息。

### 浏览器验证

- 打开 `http://127.0.0.1:8080/index.html`。
- 老板登录后看板仍显示 38 店总览。
- 店长 `rg1` 登录后，只看到本店趋势和相关数据。
- 数据录入页上传带不同月份的 CSV，出现月份冲突提示。
- 截图识别未配置时显示明确降级文案。
- 报销、工资、巡店弹窗支持 Esc 关闭。

### 构建验证

- 后端运行 `mvn -q -DskipTests package`。
- 前端内联脚本语法检查通过。
- 浏览器 console 不出现新增错误。

## 风险与缓解

风险一：旧前端函数很多，直接替换统计入口可能影响老板视角。

缓解：只在统计函数入口统一可见门店集合，老板可见范围保持全量；用老板和店长两个角色做回归截图。

风险二：助手加元信息后，旧前端可能未消费新字段。

缓解：后端响应保持兼容，新增字段只追加不破坏原有 `answer` 或文本字段。

风险三：导入月份冲突涉及前端状态，可能影响正常单月导入。

缓解：只有识别到 `sourceMonth` 且与 `pageMonth` 不一致时才进入冲突状态；无月份文件仍沿用当前页面月份但显示“文件未识别月份”提示。

风险四：Esc 关闭误关正在编辑的弹窗。

缓解：有未保存输入时先弹确认；无输入或只读预览可以直接关闭。

## 成功标准

P0 完成后，系统仍保持当前功能范围，但用户看到的数字更可信：

- 店长看不到不属于自己的经营趋势。
- AI 助手回答能解释数据来源。
- 导入不会把文件月份静默改成页面月份。
- 未配置截图识别时不误导用户。
- 常用弹窗有可靠退出方式。

这一步完成后，下一阶段可以更稳地推进角色工作台、手机端叫货、仓库采购 V1 和 AI Agent 中心。
