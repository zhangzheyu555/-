# 小程序后端能力缺口台账

## 用途与维护规则

本台账记录小程序需求在**不修改后端**前提下无法完整交付、或会造成越权/失败操作的能力缺口。

- 小程序开发发现接口缺失、状态机不支持、字段不足或权限不匹配时，必须先在此追加记录，再决定是否隐藏入口、降级展示或提交后端需求。
- 每条记录必须包含：业务角色、用户场景、接口证据、前端处理方式、后端所需改动和状态。
- 不得以本地缓存、伪造成功提示或绕过前端权限代替后端能力。
- 状态仅使用：`待后端评估`、`后端开发中`、`待小程序接入`、`已关闭`。
- 已关闭的条目保留历史，不删除；在“处理结果”补充接口版本、验证日期和小程序接入说明。

## 当前未关闭条目

共 **13** 项：仓库管理员 **9** 项，店长 **4** 项；当前状态均为 **待后端评估**。

### 仓库管理员

#### MBG-001｜采购到货差异处理

- **场景**：采购单分批到货、少收、拒收或存在到货差异。
- **限制证据**：`POST /api/warehouse/purchase-orders/{id}/receive` 要求全部明细一次提交，且每项入库数量必须等于审批数量；提交后采购单直接完成入库。
- **小程序处理**：逐项录入批次、到货/到期日期和备注；不展示“部分收货”入口。
- **需要后端提供**：行级累计收货、差异/拒收原因、`PARTIALLY_RECEIVED` 状态、未收数量继续收货及审计日志。
- **状态**：待后端评估。

#### MBG-002｜供应商维护

- **场景**：新增、编辑、停用供应商及维护联系人。
- **限制证据**：仓库概览只返回可选供应商；当前 Controller 未提供供应商维护接口。
- **小程序处理**：仅允许选择现有有效供应商。
- **需要后端提供**：供应商详情、新增、编辑、启停接口；包含联系人、电话、结算信息，并校验历史采购单关联。
- **状态**：待后端评估。

#### MBG-003｜既有物料完整编辑

- **场景**：编辑图片、单位换算、保质期、仓位、预警及部门归属。
- **限制证据**：`POST /api/warehouse/items` 虽可保存物料，但缺少安全的详情回填/局部更新契约；提交局部字段可能覆盖既有数据。
- **小程序处理**：支持新增、分类维护、启停；既有物料保持只读查看。
- **需要后端提供**：完整详情与 PATCH 契约，或明确全量编辑字段；安全保存图片、单位换算、保质期、仓位和部门归属。
- **状态**：待后端评估。

#### MBG-004｜总仓盘点与临期处置

- **场景**：总仓盘点、库存调整、报损/报废和临期处置。
- **限制证据**：盘点接口位于 `/api/operations/inventory-checks` 且归运营角色；每日报损接口面向门店。
- **小程序处理**：展示库存预警、批次和流水，不展示越权操作。
- **需要后端提供**：按仓库维度的盘点、差异确认、库存调整、临期处置、审批和操作日志接口。
- **状态**：待后端评估。

#### MBG-005｜仓库发起配送退货

- **场景**：仓库人员主动新建配送退货单。
- **限制证据**：`POST /api/warehouse/returns` 使用叫货创建权限，并校验原叫货单所属门店。
- **小程序处理**：保留退货审核、收货；不展示仓库发起入口。
- **需要后端提供**：与门店退货区分的单据、权限、库存方向和审核规则。
- **状态**：待后端评估。

#### MBG-006｜手工批次入库

- **场景**：盘盈、赠品、紧急补货等非采购单入库。
- **限制证据**：`POST /api/warehouse/stock-batches` 在当前仓库拓扑下要求先创建并审批采购单。
- **小程序处理**：只通过采购单收货录入批次。
- **需要后端提供**：入库类型、审批规则、凭证/附件、库存流水来源；否则继续以采购单为唯一入口。
- **状态**：待后端评估。

#### MBG-007｜主动分配货物

- **场景**：因缺货、库存积压或临期，仓库主动向门店均分/分配货物。
- **限制证据**：需求说明书 4.2.4 明确要求；现有接口只有门店叫货和仓间调拨，没有主动分配单或门店待收货接口。
- **小程序处理**：展示低库存、临期和积压风险，供人工判断；不伪造分配入口。
- **需要后端提供**：分配策略、主动分配单、按门店库存排序/均分、实收差异、门店确认入库和完整审计。
- **状态**：待后端评估。

#### MBG-008｜拣货过程留痕

- **场景**：拣货中状态、按批次拣货、拣货差异和暂缓发货。
- **限制证据**：叫货单审核后只能直接发货，缺少拣货任务、批次分配与过程状态接口。
- **小程序处理**：生成只读拣货清单和库存比对；不提前扣库存或把勾选当作业务完成。
- **需要后端提供**：拣货任务/明细、拣货状态、批次分配、拣货人、差异和时间戳；发货应基于确认的拣货结果。
- **状态**：待后端评估。

#### MBG-012｜创建采购单因版本字段无默认值失败

- **场景**：仓库管理员在小程序选择供应商、物料、数量和单价后创建采购单。
- **限制证据**：`POST /api/warehouse/purchase-orders` 在本机 `http://127.0.0.1:8080` 返回 `INTERNAL_ERROR`，请求编号为 `94ad706c12e1`。后端日志显示 MySQL 错误 `1364`：`Field 'version' doesn't have a default value`。本机表 `warehouse_purchase_order.version` 当前为 `bigint not null` 且默认值为空；`WarehouseRepository.insertPurchaseOrder` 的 INSERT 未写入该字段。采购入库接口 `POST /api/warehouse/purchase-orders/{id}/receive` 也在请求 `c50df6ed2b87` 失败：`WarehouseRepository.upsertBatch` 写入 `warehouse_stock_batch` 时同样未传 `version`，而该列无默认值。
- **小程序处理**：请求字段与后端 `WarehousePurchaseOrderRequest` 契约一致，保留失败提示及幂等请求编号；不伪造采购单创建成功，也不改写数据库。
- **需要后端提供**：新增 Flyway 修复迁移，为 `warehouse_purchase_order.version`、`warehouse_stock_batch.version` 恢复默认值 `0`，或由仓储层 INSERT/UPSERT 显式写入初始版本 `0`；补充真实 MySQL 下的采购单创建、审批、入库、幂等重试与库存批次写入回归测试。
- **状态**：待后端评估。

### 店长

#### MBG-009｜门店独立盘存

- **场景**：本店批次/效期盘点、差异调整。
- **限制证据**：需求说明书 4.4 要求独立盘存；现有盘点权限仅授予运营角色，数据模型也缺少批次和到期字段。
- **小程序处理**：展示本店库存、效期提醒、叫货和收货，不展示无权限盘存入口。
- **需要后端提供**：店长本店范围的创建/提交权限；批次、到期、账面/实盘数、差异原因与调整流水，复核权限保持独立。
- **状态**：待后端评估。

#### MBG-010｜门店自采入库与成本记录

- **场景**：语音/文字录入自采并进入库存和成本账。
- **限制证据**：需求说明书 4.5 要求该能力；当前没有自采单、识别确认、库存入库或成本入账接口。
- **小程序处理**：不提供会造成账实不一致的自采入库入口。
- **需要后端提供**：自采单、识别草稿与人工确认、门店库存入库、成本记录、附件和审批/审计接口。
- **状态**：待后端评估。

#### MBG-011｜门店叫货提交因版本字段无默认值失败

- **场景**：店长在小程序选择物料并提交门店叫货单。
- **限制证据**：`POST /api/warehouse/requisitions` 返回 `INTERNAL_ERROR`。后端日志 `requestId=8bb725d32e6f` 显示，执行 `insert into store_requisition(...)` 时 MySQL 返回错误码 `1364`：`Field 'version' doesn't have a default value`。`V43__multi_warehouse_topology.sql` 最初将 `version` 定义为 `bigint not null default 0`，但 `V56__add_chinese_column_comments.sql` 将其改为 `bigint not null` 且未保留默认值；当前 `WarehouseRepository.insertRequisition` 的 INSERT 也没有显式写入 `version`。
- **小程序处理**：请求字段与后端 `WarehouseRequisitionRequest` 契约一致，保留当前失败提示和幂等请求编号，不伪造成功、不自动改写数据库。
- **需要后端提供**：新增 Flyway 修复迁移恢复 `store_requisition.version` 的默认值，或由仓储层 INSERT 显式写入初始版本；补充真实 MySQL 下的叫货创建回归测试，并验证幂等重试不会产生重复单据。
- **状态**：待后端评估。

#### MBG-013｜店长报损写权限与物料、记录读权限冲突

- **场景**：店长在小程序选择本店物料并提交每日报损，随后查看处理状态；督导在小程序加载待复核报损。
- **限制证据**：店长具备 `daily_loss.create`，但 `GET /api/daily-loss/items` 与 `GET /api/daily-loss/records` 均调用 `requireDailyLossRead`；`V77__restrict_daily_loss_read_and_export_to_finance.sql` 又明确删除店长、督导的 `daily_loss.read`，因此两接口对店长返回 403，督导虽有 `daily_loss.review` 也无法加载待复核记录。此外，`/items` 返回 `loss_item_config.id`，而旧版 `POST /records` 按 `warehouse_item.id` 查询物料，两套 ID 契约不一致。
- **小程序处理**：店长提交表单改用已有权限的本店库存概览作为物料来源，以 `warehouse_item.id` 提交；店长不再请求或显示无权访问的报损记录筛选，提交结果使用明确弹窗。财务、老板仍可按权限查询记录；督导无法加载复核列表时显示能力限制，不伪造数据。
- **需要后端提供**：拆分“提交所需物料读取”与财务报损查询权限，或提供仅返回店长本店可报损物料的接口；统一物料 ID 契约；为督导提供授权门店的待复核列表/详情接口，同时保持财务汇总查询边界。
- **状态**：待后端评估。
