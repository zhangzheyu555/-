# 员工信息管理（员工档案）设计文档

> 版本：v1.0　日期：2026-07-17　状态：待评审
> 项目：企迈链接销售汇总，物料损耗，营业额（Spring Boot 3 + Vue3 + MySQL 8）

---

## 1. 背景与目标

公司全部门店员工（现约 110 人，见桌面《门店人员信息.xlsx》）的档案目前只存在 Excel 里，
系统内的 `employee` 表只有工资模块用的基础字段，且**只读**（无增删改界面，数据靠种子脚本导入）。

本设计要做到：

1. **一次性导入**全公司员工信息（以《门店人员信息.xlsx》为源）；
2. 系统内可**增、删、改、查**员工档案；
3. 档案覆盖：**健康证（办理/到期日期）、生日、入职日期**、身份证、合同、晋升节点等全部现有 Excel 字段；
4. **自动生成员工登录账号**：以所在门店店长账号为前缀顺延编号
   ——店长账号是 `ruguo1`，该店员工账号依次为 `ruguo1-1`、`ruguo1-2`、`ruguo1-3`…

## 2. 现状盘点（设计基于现有代码扩展，不另起炉灶）

| 现有资产 | 说明 | 本设计如何利用 |
| --- | --- | --- |
| `employee` 表（V15 建，V24 加 data_source） | 已有 id/门店/姓名/电话/岗位/用工类型/底薪/状态/入职日期/备注 | **扩展列**，不建新表；工资、考试模块已引用它，改造后自动受益 |
| `auth_user` 表 | 登录账号（id/username/password_hash/display_name/role/store_id/enabled） | 员工账号写入此表，`role=EMPLOYEE`，通过新列与 employee 关联 |
| `GET /api/employees` | 只读列表（EmployeeController） | 保留并扩展返回字段；新增写接口 |
| 员工工作台（6ea9226 新增） | EMPLOYEE 角色登录后的工作台、我的资料页 | 员工用生成的账号登录后可看到自己的档案 |
| `store_branch` 表 | 门店主数据（目前 6 家种子门店） | 员工挂靠门店；导入前需补齐缺失门店 |
| 权限体系（V39/V61 角色模板+数据范围） | BOSS 全部、店长限本店 | 员工档案接口沿用同一套数据范围控制 |

## 3. 数据模型设计

### 3.1 `employee` 表扩展（新迁移 V66__employee_profile_fields.sql）

```sql
alter table employee
  add column birthday varchar(20) null comment '生日(月.日，Excel 无年份，如 4.14)' after hire_date,
  add column id_card_no varchar(30) null comment '身份证号码' after birthday,
  add column health_cert_issue_date date null comment '健康证办理日期' after id_card_no,
  add column health_cert_expire_date date null comment '健康证到期日期' after health_cert_issue_date,
  add column contract_sign_text varchar(60) null comment '合同签署时间(Excel 原文，含区间写法)' after health_cert_expire_date,
  add column regular_date date null comment '转正时间' after contract_sign_text,
  add column trainer_date date null comment '训练员转正时间' after regular_date,
  add column shift_leader_date date null comment '领班时间' after trainer_date,
  add column manager_date date null comment '店长转正时间' after shift_leader_date,
  add column auth_user_id bigint null comment '关联登录账号' after manager_date,
  add unique key uk_employee_auth_user (auth_user_id),
  add constraint fk_employee_auth_user foreign key (auth_user_id) references auth_user(id);

create index idx_employee_health_expire on employee (tenant_id, health_cert_expire_date);
```

字段口径说明：

- **生日存字符串**：Excel 只有「4.14」这种月.日，无年份，强转 date 会造假数据；
  页面按「MM.DD」展示，到生日提醒场景再按月日匹配。
- **contract_sign_text 存原文**：Excel 里有「2025.8.22-20」这类区间/简写，先保真存储，
  后续规范后可加结构化列。
- **身份证号**：Excel 里部分是截断的 12 位（如 `421003198104`），照原样存，页面标注「不完整」；
  接口返回默认**脱敏**（前 6 后 2，中间打星），仅 BOSS 可看全量。
- 迁移编号 V66 起（V63-65 已被企迈占用），同时提供 `db/migration-h2` 等价版本。

### 3.2 账号关联关系

```
auth_user (1) ←── auth_user_id ── (0..1) employee
```

- 员工可以没有账号（先建档、后开号）；一个账号只对应一个员工。
- 删除员工档案时账号同步 `enabled=0`（软禁用，不物理删，保审计）。

## 4. 员工账号自动生成规则（核心需求）

### 4.1 规则

```
员工账号 = {所在门店店长账号}-{序号}     序号从 1 开始，逐人 +1
```

示例：`ruguo1` 店的员工依次是 `ruguo1-1`、`ruguo1-2`、`ruguo1-3`…

### 4.2 生成算法

1. 取员工所属门店 `store_id`，查该店 `role=STORE_MANAGER` 的账号 username 作为前缀
   （一店多店长时取最早创建的；无店长账号时**拒绝开号**并提示先配店长）。
2. 序号 = 该前缀现有最大序号 + 1：
   `select max(cast(substring_index(username,'-',-1) as unsigned)) from auth_user where username like '{prefix}-%'`
3. **序号只增不复用**：离职删档后编号空着，避免新人继承旧人账号造成工资/考试记录串档。
4. 并发安全：生成在一个事务里完成，`auth_user.username` 已有唯一约束兜底，撞号则重试。
5. 新账号默认值：`role=EMPLOYEE`、`store_id=员工门店`、`display_name=员工姓名`、
   初始密码统一 `Emp@12345`（bcrypt 入库），首次登录强制改密（复用现有改密接口）。

### 4.3 与现状的衔接

当前库里店长账号是 `rg1`、`bw2`、`rx3` 这种简写，你的例子用的是 `ruguo1`。
两种都兼容——规则只认「该店 STORE_MANAGER 的 username」，店长叫什么前缀就是什么。
如果要统一改成 `ruguo1` 风格，先在账号权限页改店长 username，再开员工号即可。

## 5. 接口设计（`/api/employees`，沿用现有 Controller 扩展）

| 方法 | 路径 | 说明 | 权限 |
| --- | --- | --- | --- |
| GET | `/api/employees?storeId=&status=&keyword=&healthExpireInDays=` | 分页列表+筛选（健康证 N 天内到期） | BOSS/OPERATIONS 全部，店长限本店 |
| GET | `/api/employees/{id}` | 单个详情（身份证脱敏） | 同上；员工本人可查自己 |
| POST | `/api/employees` | 新增（可选 `createAccount:true` 同时开号） | BOSS、店长(本店) |
| PUT | `/api/employees/{id}` | 修改档案 | BOSS、店长(本店) |
| DELETE | `/api/employees/{id}` | 删除=状态改「离职」+禁用账号（软删） | BOSS、店长(本店) |
| POST | `/api/employees/{id}/account` | 为已有员工补开账号（返回账号+初始密码） | BOSS、店长(本店) |
| POST | `/api/employees/import` | 上传 xlsx 导入，返回导入报告 | 仅 BOSS |
| GET | `/api/employees/export` | 导出 CSV（当前筛选范围） | BOSS/OPERATIONS/店长(本店) |

所有写操作记入现有操作日志（operation log）。

## 6. Excel 导入设计（一次性初始化 + 之后增量可重跑）

### 6.1 列映射（源：《门店人员信息.xlsx》Sheet1，第 2 行为表头，112 行数据）

| Excel 列 | 目标字段 | 清洗规则 |
| --- | --- | --- |
| 门店 | store_id | 经「门店别名对照」映射（见 6.2） |
| 姓名 | name | 去空格 |
| 职位 | position | 原样（店长/训练员/领班/店员…） |
| 入职时间 | hire_date | `2022.5.18` → date；只有年月的补 01 日 |
| 转正时间 | regular_date | 同上 |
| 训练员转正 | trainer_date | 同上 |
| 领班 | shift_leader_date | 同上 |
| 店长转正 | manager_date | 同上 |
| 生日 | birthday | 原样存「月.日」字符串 |
| 联系方式 | phone | 去空格，校验 11 位，不合法进报告 |
| 身份证号码 | id_card_no | 原样存；非 18 位标记「不完整」 |
| 健康证办理日期 | health_cert_issue_date | 同日期规则 |
| 健康证到期日期 | health_cert_expire_date | 同日期规则 |
| 合同签署时间 | contract_sign_text | 原文存字符串 |

### 6.2 门店映射

Excel 用简称（长大、吾悦…），`store_branch` 目前只有 6 家种子店。导入前：

1. 先跑「预检」：列出 Excel 里出现的全部门店简称 × 现有 store_branch 的对照缺口；
2. 缺的门店由 BOSS 在门店管理页补建（或导入界面里现场建）；
3. 对照关系存 `store_branch.remark` 或新列 `alias`，重复导入时自动命中。

### 6.3 幂等与冲突

- 天然键 = 表内现有唯一键 `(tenant_id, store_id, name)`；
- 再次导入时同键记录走 **update**（Excel 为准）而不是报错，实现「改了 Excel 重导即可」；
- 同店同名两人的极端情况：导入报告标冲突，人工在页面上改其一姓名（如「张三(大)」）后重导。
- 导入报告返回：新增 N、更新 N、跳过 N + 每条失败原因（门店未映射/日期解析失败/手机号非法）。
- 导入**不自动开账号**（避免一次生成上百个账号失控），开号在页面上按店批量执行。

## 7. 前端设计

### 7.1 新页面「员工档案」（`/staff`，菜单挂在运营中心组）

- 可见角色：BOSS、OPERATIONS 全部门店；STORE_MANAGER 仅本店（复用 allowedRoles + 数据范围）。
- 列表列：姓名｜门店｜职位｜账号(无则「未开号」按钮)｜入职日期｜生日｜健康证到期日｜状态。
- **健康证到期高亮**：已过期红色、30 天内到期黄色；顶部筛选「30 天内到期」一键查看。
- 工具栏：搜索（姓名/手机）、门店筛选、状态筛选、「导入 Excel」（仅 BOSS）、「导出 CSV」。
- 行操作：编辑（弹窗表单，字段同 §6.1）、开账号（弹窗显示生成的账号+初始密码，可复制）、离职。
- 批量开号：勾选多人 → 按各自门店规则连续生成，结果表格展示可导出（发给各店长）。

### 7.2 员工侧

员工用 `ruguo1-1` 账号登录 → 进入现有「员工工作台」→「我的资料」页
读 `GET /api/employees/me`（按 auth_user_id 反查档案），只读展示本人档案，
发现信息有误联系店长修改。

## 8. 权限与安全

1. 身份证、手机号在非 BOSS 角色的响应里脱敏（`421003********`、`150****4899`）。
2. 员工档案属敏感个人信息：导出 CSV 动作记操作日志（谁、何时、导了哪个范围）。
3. 初始密码不落库明文、不写日志，只在开号响应里一次性返回。
4. 数据范围沿用现有 DataScope：店长的一切读写强制 `store_id=本店`（后端校验，不信前端）。

## 10. 风险与待确认

1. **Excel 数据脏**：日期格式至少 4 种写法、身份证多数截断、生日无年份——导入按「保真存储+报告标记」处理，不猜测补全；建议导入后安排各店长核一遍本店数据。
2. **门店主数据缺口**：系统 6 家种子店 vs 实际 30+ 家店，需要先补门店（这也是企迈门店映射一直没做的同一个缺口，可一并解决）。
3. **待确认**：① 员工账号初始密码用 `Emp@12345` 还是别的？② 离职员工档案保留多久？③ 健康证是否需要传照片附件（本版只有日期，附件可作 v1.1，复用系统现有 attachment 机制）？
