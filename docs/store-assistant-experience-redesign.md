# 门店经营助手体验改版设计文档

## 1. 改版目标

本次只改正式前端 `frontend-vue` 的门店经营助手，并复用现有后端经营数据与 AI 分析接口。目标不是增加装饰，而是让老板在打开页面后更快看清经营结论、更早获得有效反馈，并能直接把行动建议转成待办。

需要解决三个明确问题：

1. **工作区太小**：回答被限制在窄列和固定内嵌区域内，页面右侧留白明显，长分析需要频繁滚动。
2. **22 秒等待没有反馈**：当前请求在数据库查询和 AI 分析全部结束后才显示结果，用户在等待期间看不到已完成的经营数据。
3. **回答层级混乱**：技术信息、经营事实、结论、风险、原因和行动建议同时争夺注意力，卡片嵌套过多，阅读路径不清晰。

## 2. 设计原则

- **先事实，后分析**：先返回并展示 MySQL 经营数据，再在同一回答中补充 AI 判断。
- **先结论，后依据，最后行动**：默认阅读顺序固定为“核心判断 → 关键发现/风险 → 可能原因 → 本周行动”。
- **等待过程可理解**：显示真实阶段和已等待时间，不使用无法验证的百分比进度。
- **一屏完成主要任务**：桌面端充分使用可用宽度，输入区固定在工作区底部，分析正文获得主要高度。
- **业务语言优先**：模型、请求号等技术信息降级到次要元数据，不与结论同级展示。
- **保留权限和数据边界**：不引入前端业务数据存储，不改变后端权限与 MySQL 数据源规则。

## 3. 信息架构

### 3.1 页面结构

```text
页面标题 + 清空记录
分析范围 / 月份 / 核心经营指标（单行紧凑栏）
┌──────────────────────── 助手工作区 ────────────────────────┐
│ 问题                                                         │
│ 经营数据（第一阶段立即出现）                                  │
│ 核心判断                                                     │
│ 关键发现                 风险与异常                           │
│ 可能原因                                                     │
│ 本周行动建议（可加入待办）                                   │
│ 数据限制与分析信息（弱化）                                   │
├─────────────────────────────────────────────────────────────┤
│ 回答方式 / 服务状态                                           │
│ 输入问题                                             发送     │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 回答层级

| 优先级 | 内容 | 展示方式 |
| --- | --- | --- |
| P0 | 核心判断 | 高对比结论区，字号和行距优先 |
| P1 | 关键发现、风险 | 双栏信息区；窄屏自动单栏 |
| P1 | 本周行动建议 | 有序步骤，负责人、期限、验收指标清晰分组 |
| P2 | 可能原因 | 明确标注“需业务核实”，避免把推测当事实 |
| P3 | 模型、耗时、请求号、数据限制 | 置于回答末尾的次要信息区 |

## 4. 响应速度方案

### 4.1 两阶段返回

对于“自动”和“深度分析”模式，前端将一次交互拆成两个可见阶段：

1. 先以 `LOCAL` 模式查询经营数据并立即渲染事实结果。
2. 读取第一阶段返回的 `snapshotId`，再携带用户原始模式请求分析；后端复用同一只读快照，避免分析口径漂移。

“查数据”模式仍只发起一次本地查询。

### 4.2 等待反馈

- 数据查询阶段显示“正在读取经营数据”。
- 数据返回后立即展示指标，并在其下方显示“经营数据已就绪，AI 正在分析”。
- 显示实际已等待秒数；超过约 8 秒后文案切换为“正在整理原因与行动建议”。
- 不显示虚假百分比，不承诺具体完成时间。

### 4.3 模型与超时策略

- “自动”模式使用 `deepseek-v4-flash` 和最多 700 tokens，优先保证日常问答速度。
- “深度分析”使用 `DEEPSEEK_MODEL` 配置的模型，允许管理员按需切换 Pro。
- 单次完整分析（包括传输重试、空响应降级和质量修复）的硬预算为 30 秒；降级重试只能使用剩余预算。
- 快速与深度分析使用不同缓存键，避免快速答案被误当成深度答案。

### 4.4 性能验收口径

- 本地经营数据的首次可见时间不再被 AI 的 22 秒耗时阻塞。
- AI 总耗时仍以真实上游模型响应为准；页面明确显示耗时，不把网络等待伪装成前端卡顿。
- 相同问题仍保留后端缓存能力。

## 5. 视觉与交互规范

### 5.1 尺寸

- 桌面回答宽度使用工作区全部可用宽度，正文内容最大阅读宽度控制在约 `1180px`。
- 工作区圆角统一为 `8px`，避免多层大圆角卡片。
- 主要正文 `15px / 1.7`，标题采用 14–18px 紧凑层级。
- 输入框和主要按钮最小高度 `46px`；移动端所有点击目标不小于 `44px`。

### 5.2 色彩

- 继续使用现有青绿色品牌色和设计令牌。
- 品牌色只用于当前模式、主要操作、结论与完成状态。
- 风险使用现有警告/危险语义色，不把整段正文染色。
- 正文保证清晰对比度，技术元数据使用次级文字色。

### 5.3 状态

- 空状态：提供三个与当前月份、门店关联的建议问题。
- 数据加载：骨架与真实阶段文案同时出现。
- AI 加载：本地数据保持可读，分析区域显示阶段反馈。
- 错误：保留已成功返回的经营数据，只对失败的 AI 阶段提供“重新分析”。
- 全屏：保留现有展开能力，并补充清晰的文字提示。

## 6. 响应式规则

- `> 1280px`：范围栏与指标同排；发现/风险双栏；回答区使用完整宽度。
- `721–1280px`：范围栏分两行；分析内容单栏；指标保持 2–4 列。
- `<= 720px`：页面自然滚动，范围栏、指标、回答和行动信息全部单列；输入区保持可见且不遮挡内容。
- 支持 `prefers-reduced-motion`，加载反馈只保留静态状态变化。

## 7. 实现范围

- 修改 `frontend-vue/src/pages/AssistantPage.vue`：两阶段请求、等待计时、回答结构与样式。
- 复用 `frontend-vue/src/api/assistant.ts` 已有的 `snapshotId` 请求字段，不新增浏览器业务数据存储。
- 增补或调整助手 E2E 测试，覆盖本地数据先出现、AI 后完成、桌面和移动端布局。
- 不修改数据库结构，不新增 Flyway 迁移。

## 8. 验收标准

- 1920×1080 下回答正文不再局限于约 900px 左侧窄列，工作区高度明显增加。
- AI 请求期间，用户能先看到本次问题对应的真实经营数据。
- 完整回答的首屏顺序固定为核心判断、发现/风险、行动建议；技术信息不再占据标题主视觉。
- 390px 移动端无横向滚动，按钮和输入区可正常操作。
- Vue 类型检查与生产构建通过；相关 E2E 测试通过。
