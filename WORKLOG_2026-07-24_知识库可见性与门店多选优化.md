# WORKLOG — 知识库可见性与门店多选优化

日期：2026-07-24

## 结论

已完成知识库普通账号可见性、上传发布语义、可查看资料目录、门店复选多选、会话刷新和权限边界优化。

本次复用现有知识库资料、角色范围、门店范围、用户数据范围及生产 V76（H2 对应 V74）权限配置；未新增或修改 Flyway 迁移。

## 根因

1. 原上传接口始终写入 `DRAFT`，“全企业”仅表示范围，不表示发布；前端没有把保存草稿与发布区分。
2. 普通账号页面只有搜索，没有已发布资料目录，也没有进入、手动、聚焦刷新机制，因此新发布资料缺少可发现入口。
3. 菜单依赖登录会话内的有效权限，但前端没有在窗口重新获得焦点时更新 `/api/auth/me`。
4. 指定门店使用自由文本拆分，容易提交错误编号，也无法展示门店状态和范围。
5. 通用兼容策略把督导视为全门店；知识库的管理、资料目录、搜索和下载若直接复用，会绕过督导已配置的 STORE 范围。
6. 账号创建和授权页面也把督导当作固定 `ALL`，导致已选择的门店列表无法真实写入或回显，知识库专用范围没有可靠配置入口。

## 实施内容

### 后端

- 上传接口增加可选 `publishNow`，默认仍为草稿，保持兼容。
- 上传、解析、分段、索引、两次审计和立即发布位于同一事务；任一发布环节失败会整体回滚。
- 新增只读接口 `GET /api/knowledge-base/documents/available`。
- 可查看目录在 SQL 中完成租户、`PUBLISHED`、TENANT/ROLE/STORE 范围过滤并限制 500 条，不向前端下发越权资料。
- 下载先完成元数据及范围授权，再读取原文件 BLOB；跨租户已知资料、跨门店及无权限读取返回 403。
- `/api/stores?knowledgeBaseScope=true` 复用原门店接口，返回知识库管理者可配置的门店。
- 督导知识库管理、资料目录、搜索和下载统一使用 `user_data_scope` 中持久化的 STORE 配置范围；旧 `ALL` 配置按无知识库门店范围处理，避免默认放大权限。
- 督导账号创建、基础范围变更和统一授权保存会把真实门店 ID 同步到现有 `user_store_scope` 与 `user_data_scope`；账号管理响应回显真实配置，而普通经营模块继续保留既有督导全门店兼容行为。
- 兼容的旧授权接口同样同步两处现有范围数据；统一授权接口若收到账号门店范围与 STORE 数据范围不一致的请求，会在写入前拒绝，消除双真源。
- 只有 BOSS、SUPERVISOR 且拥有 `knowledge_base.manage` 才可管理；六类正式角色拥有 `knowledge_base.search` 时均可检索和查看目录，个人 `DENY` 优先。

### 前端

- 增加“上传并发布”和“仅保存草稿”，成功文案以服务端返回的 `status` 为准。
- 全企业上传并发布、全企业草稿“立即发布”均二次确认。
- 增加“我可查看的资料”，支持进入页面、手动、窗口聚焦及重新可见时刷新。
- 发布、下架后的目录强制刷新不会被先前进行中的旧请求吞掉。
- 管理区同时校验正式角色和权限；普通账号不再请求无关的门店列表。
- 删除 `storeScopesText` 和手工拆分逻辑，改为 `storeScopes: string[]`。
- 门店选择器展示名称、编号、区域、状态，支持搜索、全选当前结果、清空和已选数量，提交值为 `StoreInfo.id`。
- 会话在窗口重新获得焦点或重新可见时刷新 `/api/auth/me`，更新菜单和本地会话；资料内容更新不要求重新登录。
- 账号管理中督导不再显示为固定全部门店，可通过复选框创建、编辑并保存知识库门店范围；旧 `ALL` 授权会在界面中安全归一为指定门店或无范围。
